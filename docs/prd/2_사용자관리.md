## 1. 배경 및 목적

### 1.1 문제 정의

API 모니터링 서비스가 성장함에 따라 관리자가 전체 사용자 현황을 중앙에서 파악하고 제어할 수 있는 전용 관리 도구가 필요하다. 현재 관리자는 직접 데이터베이스에 접근하지 않고는 사용자 계정 상태를 변경하거나 서비스 공지를 발송하는 수단이 없다.

### 1.2 비즈니스 목표

- 관리자가 사용자 계정 생명주기(활성/정지/탈퇴)를 UI 없이 API 수준에서 제어할 수 있도록 한다.
- 이상 사용자를 신속하게 식별하고 제재하여 서비스 품질을 보호한다.
- 전체 또는 특정 사용자에게 운영 공지를 발송하여 커뮤니케이션 채널을 단일화한다.
- Admin API를 User API와 물리적으로 분리하여 권한 경계를 명확히 한다.

### 1.3 성공 지표 (KPI)

| 지표 | 목표 |
|------|------|
| 사용자 목록 조회 응답 시간 | p95 기준 300ms 이하 |
| 상태 변경 오류율 | 1% 미만 |
| 공지 발송 실패율 | 0.1% 미만 |
| Admin API 비인가 접근 차단율 | 100% |

## 2. 사용자 및 이해관계자

### 2.1 타겟 사용자 (관리자 페르소나)

| 구분 | 특성 |
|------|------|
| **SUPER_ADMIN** | 모든 권한 보유. 사용자 관리, 공지 발송, 관리자 계정 관리, 역할/권한 관리 가능 |
| **ADMIN** | user, post, notification 범위 권한 보유. 사용자 관리 및 공지 발송 가능. 관리자 계정·역할 관리 불가 |

### 2.2 사용자 스토리

| ID | 사용자 스토리 |
|----|--------------|
| US-001 | 나는 관리자로서, loginId와 password로 Admin API에 로그인하여 Access Token을 발급받고 싶다. |
| US-002 | 나는 관리자로서, 이름·이메일·상태로 사용자를 검색하고 페이지 단위로 목록을 조회하고 싶다. |
| US-003 | 나는 관리자로서, 특정 사용자의 가입일, 수정일, 상태 등 상세 정보를 단건으로 조회하고 싶다. |
| US-004 | 나는 관리자로서, 약관 위반 사용자의 계정을 INACTIVE 상태로 변경하여 서비스 이용을 제한하고 싶다. |
| US-005 | 나는 관리자로서, 탈퇴 처리된 사용자 데이터를 Soft Delete로 보존하면서 서비스 접근을 완전히 차단하고 싶다. |
| US-006 | 나는 관리자로서, 전체 사용자 또는 특정 사용자에게 서비스 공지(시스템, 게시글, 댓글, 사용자 관련)를 발송하고 싶다. |
| US-007 | 나는 관리자로서, 발송한 공지 목록을 유형·수신자·키워드로 필터링하여 조회하고 싶다. |

## 3. 제품 범위

### 3.1 포함 범위 (In Scope)

- Admin 로그인 / 로그아웃 / Access Token 재발급
- 사용자 목록 조회 (검색 + Offset 페이지네이션)
- 사용자 단건 상세 조회
- 사용자 계정 상태 변경 (ACTIVE / INACTIVE)
- 사용자 강제 탈퇴 (Soft Delete, 상태 WITHDRAWN 설정)
- 공지 발송 (전체 또는 특정 사용자 대상)
- 공지 목록 조회 (필터링 + Offset 페이지네이션)

### 3.2 제외 범위 (Out of Scope)

- Admin 계정 CRUD (관리자 계정 생성·수정·삭제는 별도 PRD로 관리)
- 역할(Role) / 권한(Permission) 관리
- 사용자 모니터 현황 조회 (별도 모니터 도메인 PRD로 분리)
- 공지 수정 / 삭제
- 이메일·SMS 등 외부 발송 채널 연동 (현재는 DB 저장 방식의 인앱 공지)
- 관리자 행위 감사 로그 UI

### 3.3 향후 고려사항

- 공지 수정 및 삭제 기능
- 관리자 행위에 대한 별도 감사 로그 뷰
- 전체 사용자 공지 발송 시 배치 처리 및 발송 상태 추적
- 사용자별 보유 모니터 수, 마지막 로그인 등 집계 데이터 노출 (현재 User 모델에 미포함)

## 4. 기능 요구사항

### FR-001: Admin 로그인

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-001 |
| **기능명** | Admin 로그인 |
| **엔드포인트** | `POST /api/v1/auth/signin` |
| **인증** | `@Public()` — 인증 불필요 |
| **권한** | 없음 |
| **우선순위** | P0 |

**설명**

관리자는 `loginId`와 `password`를 제출하여 로그인한다. 인증 성공 시 Access Token(응답 바디)과 Refresh Token(HttpOnly Cookie)을 발급한다. 비밀번호는 bcrypt로 해시 비교한다.

**요청 DTO**

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| `loginId` | string | Y | 관리자 로그인 ID |
| `password` | string | Y | 비밀번호 |

**응답 DTO**

| 필드 | 타입 | 설명 |
|------|------|------|
| `accessToken` | string | JWT Access Token (TTL: 1시간) |
| `admin.id` | number | 관리자 ID |
| `admin.loginId` | string | 로그인 ID |
| `admin.email` | string | 이메일 |
| `admin.name` | string | 이름 |
| `admin.phone` | string | 전화번호 |
| `admin.createdAt` | Date | 계정 생성일 |
| `admin.updatedAt` | Date \| null | 최근 수정일 |

Refresh Token은 응답 바디에 포함되지 않으며 `refreshToken` HttpOnly Cookie로만 전달된다.

**수용 기준**

```
Given 유효한 loginId와 password를 제출할 때
When POST /api/v1/auth/signin 요청 시
Then HTTP 200과 함께 accessToken 및 admin 정보를 응답하고
     refreshToken이 HttpOnly, Secure, SameSite=Strict Cookie로 설정된다.

Given 존재하지 않는 loginId를 제출할 때
When POST /api/v1/auth/signin 요청 시
Then HTTP 404 USER_ERROR_NOT_FOUND를 응답한다.

Given 올바르지 않은 password를 제출할 때
When POST /api/v1/auth/signin 요청 시
Then HTTP 401 AUTH_ERROR_PASSWORD_NOT_MATCHED를 응답한다.

Given isDeleted = true인 관리자 계정으로 로그인 시도 시
When POST /api/v1/auth/signin 요청 시
Then HTTP 404 USER_ERROR_NOT_FOUND를 응답한다.
```

### FR-002: Admin 로그아웃

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-002 |
| **기능명** | Admin 로그아웃 |
| **엔드포인트** | `DELETE /api/v1/auth/signout` |
| **인증** | JWT Access Token 필요 |
| **권한** | 없음 (인증된 관리자 본인) |
| **우선순위** | P0 |

**설명**

Refresh Token을 Cookie 또는 Authorization 헤더에서 추출하여 Redis에서 삭제(무효화)한 후 Cookie를 제거한다. 서버 측 토큰 삭제가 성공한 후에만 Cookie를 제거하여 재시도 가능성을 보장한다.

**수용 기준**

```
Given 유효한 Refresh Token Cookie를 가진 관리자가
When DELETE /api/v1/auth/signout 요청 시
Then Redis에서 해당 jti의 Refresh Token을 삭제하고
     HTTP 204를 응답하며 refreshToken Cookie를 제거한다.

Given Refresh Token Cookie가 없는 상태에서
When DELETE /api/v1/auth/signout 요청 시
Then HTTP 401 AUTH_ERROR_MISSING_REFRESH_TOKEN을 응답한다.
```

### FR-003: Access Token 재발급

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-003 |
| **기능명** | Access Token 재발급 |
| **엔드포인트** | `POST /api/v1/auth/token/refresh` |
| **인증** | `JwtRefreshGuard` + `@Public()` |
| **권한** | 없음 |
| **우선순위** | P0 |

**설명**

Redis에 저장된 Refresh Token과 요청 토큰의 bcrypt 비교를 통해 유효성 검증 후, 기존 토큰 삭제 및 새 Access Token / Refresh Token 쌍을 발급한다 (Token Rotation 방식).

**수용 기준**

```
Given 유효하고 Redis에 존재하는 Refresh Token을 가진 관리자가
When POST /api/v1/auth/token/refresh 요청 시
Then 기존 Refresh Token을 Redis에서 삭제하고
     새 Access Token(응답 바디)과 새 Refresh Token(Cookie)을 발급한다.

Given Redis에 존재하지 않는 Refresh Token으로 요청 시
Then HTTP 401 AUTH_ERROR_MISSING_REFRESH_TOKEN을 응답한다.

Given 저장된 토큰과 bcrypt 비교 불일치 시
Then HTTP 401 AUTH_ERROR_INVALID_REFRESH_TOKEN을 응답한다.
```

### FR-004: 사용자 목록 조회

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-004 |
| **기능명** | 사용자 목록 조회 |
| **엔드포인트** | `GET /api/v1/users` |
| **인증** | JWT Access Token 필요 |
| **권한** | `user:read` |
| **우선순위** | P0 |

**설명**

Offset 방식의 페이지네이션으로 사용자 목록을 반환한다. 이름(부분 일치), 이메일(부분 일치), 상태(UserStatus enum)로 필터링 가능하다. `isDeleted = false` 조건이 기본 적용되며, Soft Delete된 사용자는 목록에서 제외된다.

**요청 쿼리 파라미터**

| 파라미터 | 타입 | 필수 | 기본값 | 설명 |
|---------|------|------|--------|------|
| `page` | number | N | 1 | 페이지 번호 |
| `size` | number | N | 20 | 페이지 크기 |
| `order` | `asc` \| `desc` | N | `desc` | 정렬 방향 (id 기준) |
| `name` | string | N | - | 이름 부분 일치 검색 |
| `email` | string | N | - | 이메일 부분 일치 검색 |
| `status` | UserStatus | N | - | `ACTIVE` \| `INACTIVE` \| `WITHDRAWN` |

**응답 DTO**

```
OffsetResponseDto<AdminUserResponseDto>
  data: AdminUserResponseDto[]
  meta:
    page: number
    totalCount: number
```

`AdminUserResponseDto` 필드:

| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | number | 사용자 ID |
| `email` | string | 이메일 |
| `name` | string | 이름 |
| `phone` | string | 전화번호 |
| `status` | UserStatus | 계정 상태 |
| `createdAt` | Date | 가입일 |
| `updatedAt` | Date \| null | 최근 수정일 |

**수용 기준**

```
Given 인증된 관리자(user:read 권한)가
When GET /api/v1/users?page=1&size=20 요청 시
Then isDeleted=false 사용자 목록과 totalCount를 응답한다.

Given name=홍 파라미터를 포함하여 요청 시
Then name에 '홍'이 포함된 사용자만 필터링하여 응답한다.

Given status=INACTIVE 파라미터를 포함하여 요청 시
Then status가 INACTIVE인 사용자만 필터링하여 응답한다.

Given user:read 권한이 없는 관리자가 요청 시
Then HTTP 403을 응답한다.

Given 인증되지 않은 요청 시
Then HTTP 401을 응답한다.
```

### FR-005: 사용자 단건 조회

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-005 |
| **기능명** | 사용자 단건 상세 조회 |
| **엔드포인트** | `GET /api/v1/users/:id` |
| **인증** | JWT Access Token 필요 |
| **권한** | `user:read` |
| **우선순위** | P0 |

**설명**

특정 사용자 ID로 상세 정보를 조회한다. `isDeleted = false` 조건이 적용되어 Soft Delete된 사용자는 조회되지 않는다.

**경로 파라미터**

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| `id` | number | Y | 사용자 ID (ParseIntPipe 적용) |

**응답 DTO**

`AdminUserResponseDto` (FR-004 동일 구조)

**수용 기준**

```
Given 인증된 관리자(user:read 권한)가 유효한 id로 요청 시
When GET /api/v1/users/:id 요청 시
Then 해당 사용자의 상세 정보를 HTTP 200으로 응답한다.

Given 존재하지 않거나 isDeleted=true인 id로 요청 시
Then HTTP 404 USER_ERROR_NOT_FOUND를 응답한다.

Given 숫자가 아닌 id로 요청 시 (예: /users/abc)
Then HTTP 400 유효성 검증 오류를 응답한다.
```

### FR-006: 사용자 계정 상태 변경

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-006 |
| **기능명** | 사용자 계정 활성화 / 비활성화 |
| **엔드포인트** | `PATCH /api/v1/users/:id/status` |
| **인증** | JWT Access Token 필요 |
| **권한** | `user:update` |
| **우선순위** | P0 |

**설명**

특정 사용자의 계정 상태를 `ACTIVE` 또는 `INACTIVE`로 변경한다. 상태 변경 시 `updatedAt`과 CLS에서 추출한 관리자 ID를 `updatedBy`에 자동 기록한다. INACTIVE 상태의 사용자는 User API 로그인이 차단된다.

**경로 파라미터**

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| `id` | number | Y | 사용자 ID |

**요청 DTO**

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| `status` | UserStatus | Y | `ACTIVE` \| `INACTIVE` (변경할 상태) |

**수용 기준**

```
Given 인증된 관리자(user:update 권한)가 유효한 id와 status=INACTIVE로 요청 시
When PATCH /api/v1/users/:id/status 요청 시
Then 해당 사용자의 status를 INACTIVE로 변경하고
     updatedAt, updatedBy를 갱신하며
     HTTP 204를 응답한다.

Given status=ACTIVE로 요청 시
Then 해당 사용자의 status를 ACTIVE로 변경하고 HTTP 204를 응답한다.

Given 존재하지 않거나 isDeleted=true인 id로 요청 시
Then HTTP 404 USER_ERROR_NOT_FOUND를 응답한다.

Given user:update 권한이 없는 관리자가 요청 시
Then HTTP 403을 응답한다.
```

> **주의:** `WITHDRAWN` 상태로의 변경은 FR-007(강제 탈퇴)을 통해서만 수행한다. 이 엔드포인트에서 `WITHDRAWN`을 직접 설정하는 것은 허용하지 않는다.

### FR-007: 사용자 강제 탈퇴 (Soft Delete)

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-007 |
| **기능명** | 사용자 강제 탈퇴 |
| **엔드포인트** | `DELETE /api/v1/users/:id` |
| **인증** | JWT Access Token 필요 |
| **권한** | `user:delete` |
| **우선순위** | P0 |

**설명**

물리 삭제를 수행하지 않는다. 대신 아래 필드를 원자적으로 업데이트하여 논리적으로 탈퇴 처리한다.

| 필드 | 설정 값 |
|------|---------|
| `status` | `WITHDRAWN` |
| `isDeleted` | `true` |
| `deletedAt` | 현재 시각 |
| `deletedBy` | CLS에서 추출한 관리자 ID |

탈퇴 처리된 사용자는 `isDeleted = false` 기본 필터에 의해 이후 모든 조회에서 자동 제외된다.

**경로 파라미터**

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| `id` | number | Y | 사용자 ID |

**수용 기준**

```
Given 인증된 관리자(user:delete 권한)가 유효한 id로 요청 시
When DELETE /api/v1/users/:id 요청 시
Then status=WITHDRAWN, isDeleted=true, deletedAt=현재시각, deletedBy=관리자ID로 업데이트하고
     HTTP 204를 응답한다.

Given 존재하지 않거나 이미 isDeleted=true인 id로 요청 시
Then HTTP 404 USER_ERROR_NOT_FOUND를 응답한다.

Given user:delete 권한이 없는 관리자가 요청 시
Then HTTP 403을 응답한다.
```

### FR-008: 공지 발송

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-008 |
| **기능명** | 공지 발송 (전체 또는 특정 사용자) |
| **엔드포인트** | `POST /api/v1/notifications` |
| **인증** | JWT Access Token 필요 |
| **권한** | `notification:write` |
| **우선순위** | P0 |

**설명**

`userId` 필드 생략 시 전체 사용자 대상 공지로 처리된다(`userId = null`). `userId`를 지정하면 해당 사용자에게만 발송된다. `createdBy`에는 CLS에서 추출한 관리자 ID가 자동 기록된다. 현재는 DB 저장 방식의 인앱 공지이며, 외부 채널(이메일, 푸시 등) 연동은 포함하지 않는다.

**요청 DTO**

| 필드 | 타입 | 필수 | 제약 | 설명 |
|------|------|------|------|------|
| `userId` | number | N | - | 수신자 ID. 생략 시 전체 발송 |
| `title` | string | Y | 최대 100자 | 공지 제목 |
| `content` | string | Y | 최대 500자 | 공지 내용 |
| `type` | NotificationType | Y | - | `SYSTEM` \| `POST` \| `COMMENT` \| `USER` |

**NotificationType 정의**

| 값 | 설명 |
|----|------|
| `SYSTEM` | 시스템 점검, 정책 변경 등 서비스 운영 공지 |
| `POST` | 게시글 관련 공지 |
| `COMMENT` | 댓글 관련 공지 |
| `USER` | 사용자 계정 관련 공지 |

**수용 기준**

```
Given 인증된 관리자(notification:write 권한)가 userId 없이 요청 시
When POST /api/v1/notifications 요청 시
Then userId=null로 Notification 레코드를 생성하고
     createdBy에 관리자 ID를 기록하며
     HTTP 201을 응답한다.

Given userId를 포함하여 요청 시
Then 해당 userId와 연결된 Notification 레코드를 생성하고 HTTP 201을 응답한다.

Given title이 100자를 초과하거나 content가 500자를 초과하는 경우
Then HTTP 400 유효성 검증 오류를 응답한다.

Given 유효하지 않은 type 값으로 요청 시
Then HTTP 400 유효성 검증 오류를 응답한다.

Given notification:write 권한이 없는 관리자가 요청 시
Then HTTP 403을 응답한다.
```

### FR-009: 공지 목록 조회

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-009 |
| **기능명** | 공지 목록 조회 |
| **엔드포인트** | `GET /api/v1/notifications` |
| **인증** | JWT Access Token 필요 |
| **권한** | `notification:read` |
| **우선순위** | P1 |

**설명**

발송된 공지 목록을 Offset 페이지네이션으로 조회한다. `isDeleted = false` 기본 필터가 적용된다. 제목 또는 내용에 대한 OR 키워드 검색을 지원한다.

**요청 쿼리 파라미터**

| 파라미터 | 타입 | 필수 | 기본값 | 설명 |
|---------|------|------|--------|------|
| `page` | number | N | 1 | 페이지 번호 |
| `size` | number | N | 20 | 페이지 크기 |
| `order` | `asc` \| `desc` | N | `desc` | 정렬 방향 (id 기준) |
| `userId` | number | N | - | 수신자 ID 필터 |
| `type` | NotificationType | N | - | 알림 유형 필터 |
| `keyword` | string | N | - | 제목 또는 내용 부분 일치 검색 |

**응답 DTO**

```
OffsetResponseDto<NotificationResponseDto>
  data: NotificationResponseDto[]
  meta:
    page: number
    totalCount: number
```

`NotificationResponseDto` 필드:

| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | number | 공지 ID |
| `userId` | number \| null | 수신자 ID (null이면 전체 발송) |
| `adminId` | number \| null | 발송 관리자 ID |
| `title` | string | 공지 제목 |
| `content` | string | 공지 내용 |
| `type` | NotificationType | 알림 유형 |
| `createdAt` | Date | 발송 시각 |
| `createdBy` | number | 발송 관리자 ID (감사 필드) |

**수용 기준**

```
Given 인증된 관리자(notification:read 권한)가
When GET /api/v1/notifications?page=1&size=20 요청 시
Then isDeleted=false 공지 목록과 totalCount를 응답한다.

Given userId=1 파라미터를 포함하여 요청 시
Then userId=1인 공지만 필터링하여 응답한다.

Given keyword=점검 파라미터를 포함하여 요청 시
Then title 또는 content에 '점검'이 포함된 공지를 응답한다.

Given notification:read 권한이 없는 관리자가 요청 시
Then HTTP 403을 응답한다.
```

### 7.2 리스크

| 리스크 | 가능성 | 영향도 | 완화 방안 |

### 7.3 가정사항

- Admin 계정은 DB 시드를 통해 사전 생성되며, 자체 회원가입 기능은 제공하지 않는다.

- 전체 사용자 공지(`userId = null`) 수신 대상 필터링은 User API(앱 클라이언트)에서 처리한다.
- `macos darwin 25.2.0` 환경 기준으로 로컬 개발 및 Docker Compose로 테스트한다.

## 9. 부록

### 9.1 용어 정의

| 용어 | 정의 |
|------|------|
| **Soft Delete** | 데이터를 실제로 삭제하지 않고 `isDeleted=true`, `deletedAt`, `deletedBy`를 기록하여 논리적으로 삭제 처리하는 방식 |
| **RBAC** | Role-Based Access Control. 역할(Role)에 권한(Permission)을 부여하고, 관리자에게 역할을 할당하여 접근을 제어하는 방식 |
| **CLS** | Context-Local Storage (nestjs-cls). 요청 스코프 내에서 사용자 ID 등 컨텍스트 데이터를 전파하는 메커니즘 |
| **Token Rotation** | Access Token 재발급 시 기존 Refresh Token을 무효화하고 새 토큰 쌍을 발급하는 보안 방식 |
| **Offset 페이지네이션** | `page`와 `size`를 기반으로 `SKIP + TAKE`로 데이터를 조회하는 방식. `OffsetRequestDto / OffsetResponseDto` 사용 |
| **SUPER_ADMIN** | 모든 scope/action 권한을 보유한 최고 관리자 역할 |
| **ADMIN** | user, post, notification scope 권한을 보유한 일반 관리자 역할 |
| **UserStatus** | `ACTIVE`(정상), `INACTIVE`(비활성/정지), `WITHDRAWN`(탈퇴) |
| **NotificationType** | `SYSTEM`, `POST`, `COMMENT`, `USER` — 공지 유형 분류 |
| **jti** | JWT ID. 토큰을 고유하게 식별하는 UUID. Redis 키 구성 요소로 사용 |

### 9.2 에러 코드 참고

| 에러 코드 | HTTP | 설명 |
|----------|------|------|
| `USER_ERROR_NOT_FOUND` | 404 | 사용자 또는 관리자를 찾을 수 없음 |
| `AUTH_ERROR_PASSWORD_NOT_MATCHED` | 401 | 비밀번호 불일치 |
| `AUTH_ERROR_MISSING_REFRESH_TOKEN` | 401 | Refresh Token 누락 또는 Redis에 존재하지 않음 |
| `AUTH_ERROR_INVALID_REFRESH_TOKEN` | 401 | Refresh Token 검증 실패 (bcrypt 불일치) |

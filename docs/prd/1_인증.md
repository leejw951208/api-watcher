## 1. 배경 및 목적

### 1.1 문제 정의

API 모니터링 서비스(api-watcher)는 외부 API의 가용성과 응답 품질을 추적하는 시스템이다. 서비스에 접근하는 사용자의 신원을 확인하고, 인가된 사용자만 모니터링 대상 관리 및 알림 설정 등의 핵심 기능을 사용할 수 있도록 신뢰할 수 있는 인증 체계가 필요하다.

### 1.2 비즈니스 목표

- 이메일/비밀번호 기반의 표준 인증 흐름을 제공하여 사용자 진입 장벽을 최소화한다.
- Access Token + Refresh Token 이중 토큰 방식으로 보안성과 사용자 경험(UX)을 동시에 확보한다.
- 비밀번호 재설정 기능을 통해 계정 복구 수단을 제공하여 사용자 이탈을 방지한다.

### 1.3 성공 지표 (KPI)

| 지표 | 목표값 |
|------|--------|
| 회원가입 성공률 | 99% 이상 (정상 요청 기준) |
| 로그인 응답 시간 (P95) | 200ms 이하 |
| Refresh Token 갱신 실패율 | 0.1% 이하 |
| 비밀번호 재설정 이메일 발송 성공률 | 99% 이상 |

## 3. 제품 범위

### 3.1 포함 범위 (In Scope)

- 이메일/비밀번호 기반 회원가입
- 이메일/비밀번호 기반 로그인 (Access Token + Refresh Token 발급)
- 로그아웃 (Refresh Token 무효화)
- Refresh Token을 이용한 Access Token 갱신
- 비밀번호 재설정 (이메일 인증 링크 방식)

### 3.2 제외 범위 (Out of Scope)

- 소셜 로그인 (OAuth 2.0: Google, GitHub 등)
- 2단계 인증 (2FA / TOTP)
- 이메일 인증(회원가입 시 이메일 검증)
- 관리자(Admin) 인증 — Admin App은 별도 PRD로 관리
- 세션 기반 인증

### 3.3 향후 고려사항

- 소셜 로그인 연동
- 2단계 인증(TOTP) 적용
- 회원가입 이메일 인증 단계 추가
- 디바이스별 세션 관리 및 강제 로그아웃
- 앱(App) 클라이언트 지원 시 OTP 방식 비밀번호 재설정 엔드포인트 추가

## 4. 기능 요구사항

### FR-001: 회원가입

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-001 |
| **기능명** | 회원가입 |
| **우선순위** | P0 (필수) |
| **엔드포인트** | `POST /api/v1/auth/sign-up` |
| **인증 필요** | 없음 (`@Public()` 적용) |

**설명**

사용자가 이메일과 비밀번호를 제출하여 계정을 생성한다. 비밀번호는 bcrypt로 해싱한 후 저장하며, 평문은 절대 저장하지 않는다.

**요청 스펙**

```json
{
  "email": "user@example.com",
  "password": "P@ssw0rd123!"
}
```

| 필드 | 타입 | 필수 | 유효성 규칙 |
|------|------|------|------------|
| email | string | Y | 이메일 형식, 최대 255자 |
| password | string | Y | 최소 8자, 영문 대소문자·숫자·특수문자 각 1자 이상 포함 |

**응답 스펙**

- 성공: `201 Created`

```json
{
  "data": {
    "id": "uuid",
    "email": "user@example.com",
    "createdAt": "2026-03-19T00:00:00.000Z"
  }
}
```

**수용 기준 (Acceptance Criteria)**

| AC-ID | 조건 |
|-------|------|
| AC-001-1 | Given 유효한 이메일과 비밀번호를 제출하면, When 해당 이메일이 미사용 상태이면, Then 201을 반환하고 사용자 레코드가 생성된다. |
| AC-001-2 | Given 이미 가입된 이메일로 요청하면, Then 409 Conflict(AUTH_ERROR_001)를 반환한다. |
| AC-001-3 | Given 비밀번호 유효성을 위반하면, Then 400 Bad Request를 반환한다. |
| AC-001-4 | Given 회원가입이 성공하면, Then DB에 비밀번호가 bcrypt 해시값으로 저장되고 평문은 저장되지 않는다. |
| AC-001-5 | Given 회원가입이 성공하면, Then 응답 DTO에 password 필드는 포함되지 않는다. |

### FR-002: 로그인

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-002 |
| **기능명** | 로그인 |
| **우선순위** | P0 (필수) |
| **엔드포인트** | `POST /api/v1/auth/sign-in` |
| **인증 필요** | 없음 (`@Public()` 적용) |

**설명**

사용자가 이메일과 비밀번호를 제출하면 자격을 검증하고 Access Token과 Refresh Token을 발급한다. Refresh Token은 DB에 해시값으로 저장하여 재사용 여부를 검증할 수 있도록 한다.

**요청 스펙**

```json
{
  "email": "user@example.com",
  "password": "P@ssw0rd123!"
}
```

**응답 스펙**

- 성공: `200 OK`

```json
{
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIs..."
  }
}
```

**토큰 스펙**

| 항목 | Access Token | Refresh Token |
|------|-------------|---------------|
| TTL | 1시간 | 7일 |
| JWT payload | `{ id, type: 'ac', aud: 'user', jti, issuer }` | `{ id, type: 're', aud: 'user', jti, issuer }` |
| 저장 위치 | 클라이언트 메모리 | 클라이언트 스토리지(쿠키 권장) |
| DB 저장 | 저장 안 함 | bcrypt 해시값으로 저장 |

**수용 기준 (Acceptance Criteria)**

| AC-ID | 조건 |
|-------|------|
| AC-002-1 | Given 유효한 이메일/비밀번호이면, Then 200과 함께 accessToken, refreshToken을 반환한다. |
| AC-002-2 | Given 존재하지 않는 이메일이면, Then 401 Unauthorized(AUTH_ERROR_002)를 반환한다. |
| AC-002-3 | Given 비밀번호가 일치하지 않으면, Then 401 Unauthorized(AUTH_ERROR_002)를 반환한다. |
| AC-002-4 | Given 탈퇴(WITHDRAWN) 또는 비활성 상태의 계정이면, Then 401 Unauthorized(AUTH_ERROR_003)를 반환한다. |
| AC-002-5 | Given 로그인이 성공하면, Then DB의 refreshToken 컬럼이 새 토큰의 bcrypt 해시값으로 업데이트된다. |
| AC-002-6 | Given 로그인 요청이 5분 내 5회를 초과하면, Then 429 Too Many Requests를 반환한다. |

### FR-003: 로그아웃

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-003 |
| **기능명** | 로그아웃 |
| **우선순위** | P0 (필수) |
| **엔드포인트** | `POST /api/v1/auth/sign-out` |
| **인증 필요** | 있음 (Access Token, `JwtAccessGuard`) |

**설명**

현재 로그인한 사용자의 Refresh Token을 DB에서 무효화(null 처리)하여 이후 토큰 갱신이 불가능하게 한다. Access Token은 만료 시까지 유효하나 TTL이 1시간이므로 실질적 보안 영향은 제한적이다.

**응답 스펙**

- 성공: `200 OK`

```json
{
  "data": null
}
```

**수용 기준 (Acceptance Criteria)**

| AC-ID | 조건 |
|-------|------|
| AC-003-1 | Given 유효한 Access Token으로 요청하면, Then 200을 반환하고 DB의 refreshToken 컬럼이 null로 업데이트된다. |
| AC-003-2 | Given 이미 로그아웃된 사용자가 같은 Access Token으로 재요청하면, Then 200을 반환한다 (멱등성 보장). |
| AC-003-3 | Given Access Token 없이 요청하면, Then 401 Unauthorized를 반환한다. |

### FR-004: Access Token 갱신

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-004 |
| **기능명** | Access Token 갱신 |
| **우선순위** | P0 (필수) |
| **엔드포인트** | `POST /api/v1/auth/refresh` |
| **인증 필요** | 있음 (Refresh Token, `JwtRefreshGuard`) |

**설명**

클라이언트가 Refresh Token을 제출하면 DB에 저장된 해시값과 비교하여 일치 여부를 검증한다. 검증이 성공하면 새 Access Token과 새 Refresh Token을 발급하고 DB를 갱신한다(Token Rotation).

**요청 방식**

Refresh Token은 `Authorization: Bearer {refreshToken}` 헤더로 전달한다.

**응답 스펙**

- 성공: `200 OK`

```json
{
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIs..."
  }
}
```

**수용 기준 (Acceptance Criteria)**

| AC-ID | 조건 |
|-------|------|
| AC-004-1 | Given 유효하고 DB와 일치하는 Refresh Token이면, Then 200과 함께 새 accessToken, refreshToken을 반환하고 DB를 갱신한다. |
| AC-004-2 | Given 만료된 Refresh Token이면, Then 401 Unauthorized(AUTH_ERROR_004)를 반환한다. |
| AC-004-3 | Given 서명이 유효하지 않은 Refresh Token이면, Then 401 Unauthorized를 반환한다. |
| AC-004-4 | Given DB에 저장된 해시값과 일치하지 않는 Refresh Token이면 (재사용 시도), Then 401 Unauthorized(AUTH_ERROR_005)를 반환하고 DB의 refreshToken을 null로 초기화한다. |
| AC-004-5 | Given 로그아웃된 사용자(DB refreshToken = null)의 Refresh Token이면, Then 401 Unauthorized(AUTH_ERROR_005)를 반환한다. |

### FR-005: 비밀번호 재설정

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-005 |
| **기능명** | 비밀번호 재설정 |
| **우선순위** | P1 (중요) |
| **인증 필요** | 없음 (`@Public()` 적용) |

**설명**

3단계 흐름으로 구성된다.

1. **재설정 이메일 요청**: 사용자가 이메일을 제출하면 매직 링크가 포함된 이메일을 발송한다.
2. **토큰 유효성 검증**: 프론트엔드가 링크 클릭 시 토큰이 유효한지 사전 확인한다 (비밀번호 입력 폼 진입 전).
3. **비밀번호 변경**: 유효한 토큰과 새 비밀번호를 제출하여 비밀번호를 변경한다.

재설정 토큰은 원본을 이메일로 전송하고, SHA-256 해시값만 Redis에 저장한다 (Redis 탈취 시 원본 토큰 복원 불가). DB에는 저장하지 않는다.

```
[1단계] POST /api/v1/auth/password-reset/request
  이메일 제출 → 사용자 조회 (존재 시에만 내부 처리)
             → 토큰 생성(32바이트 hex) → SHA-256 해시를 Redis 저장 (TTL 30분)
             → 매직 링크 이메일 발송
             → 항상 200 반환 (계정 열거 방지)

[2단계] GET /api/v1/auth/password-reset/verify?token=abc123
  토큰 SHA-256 해시로 Redis 조회 → 유효 시 200, 만료/없음 시 400
  (토큰 소비하지 않음 — 단순 조회만)

[3단계] PATCH /api/v1/auth/password-reset
  토큰 + 새 비밀번호 제출 → Redis 조회 → 토큰 즉시 삭제 (단일 사용)
                         → 비밀번호 업데이트 → 전 기기 Refresh Token 무효화
                         → 비밀번호 변경 알림 이메일 발송
```

**5-1. 재설정 이메일 요청**

| 항목 | 내용 |
|------|------|
| **엔드포인트** | `POST /api/v1/auth/password-reset/request` |

요청 스펙:

```json
{
  "email": "user@example.com"
}
```

응답: `200 OK` (이메일 존재 여부와 관계없이 항상 동일 응답 — 계정 열거 공격 방지)

```json
{
  "data": { "message": "이메일을 확인해주세요." }
}
```

이메일 발송 내용:
- 제목: `[API Watcher] 비밀번호 재설정 안내`
- 본문: 재설정 링크 (예: `https://example.com/password-reset?token=abc123...`)
- 링크 유효 시간: 30분
- 재설정 페이지는 `Referrer-Policy: no-referrer` 헤더를 반환하여 토큰이 외부 서비스로 유출되지 않도록 한다.

**5-2. 토큰 유효성 검증**

| 항목 | 내용 |
|------|------|
| **엔드포인트** | `GET /api/v1/auth/password-reset/verify?token=abc123` |

응답:
- 유효: `200 OK`
- 만료/없음: `400 Bad Request` (`AUTH_ERROR_006`)

**5-3. 비밀번호 변경**

| 항목 | 내용 |
|------|------|
| **엔드포인트** | `PATCH /api/v1/auth/password-reset` |

요청 스펙:

```json
{
  "resetToken": "abc123...",
  "newPassword": "NewP@ssw0rd456!"
}
```

응답: `204 No Content`

**수용 기준 (Acceptance Criteria)**

| AC-ID | 조건 |
|-------|------|
| AC-005-1 | Given 가입된 이메일로 요청하면, Then 재설정 링크 이메일이 발송되고 Redis에 SHA-256 해시값이 저장된다 (TTL 30분). |
| AC-005-2 | Given 존재하지 않는 이메일로 요청하면, Then 이메일을 발송하지 않지만 응답은 200으로 동일하다 (계정 열거 방지). |
| AC-005-3 | Given 유효한 토큰으로 검증 요청하면, Then 200을 반환하고 토큰을 소비하지 않는다. |
| AC-005-4 | Given 만료되거나 없는 토큰으로 검증 요청하면, Then 400(AUTH_ERROR_006)을 반환한다. |
| AC-005-5 | Given 유효한 토큰과 새 비밀번호를 제출하면, Then 비밀번호가 변경되고 Redis에서 토큰이 즉시 삭제된다 (단일 사용). |
| AC-005-6 | Given 만료되거나 이미 사용된 토큰으로 비밀번호 변경을 요청하면, Then 400(AUTH_ERROR_006)을 반환한다. |
| AC-005-7 | Given 비밀번호 변경이 성공하면, Then 해당 사용자의 모든 Refresh Token이 무효화된다 (전 기기 강제 로그아웃). |
| AC-005-8 | Given 비밀번호 변경이 성공하면, Then 비밀번호 변경 알림 이메일이 발송된다. |
| AC-005-9 | Given 재설정 요청이 이메일 기준 3회/시간 또는 IP 기준 5회/시간을 초과하면, Then 429를 반환한다. |
| AC-005-10 | Given 비밀번호 변경 요청이 IP 기준 10회/시간을 초과하면, Then 429를 반환한다 (토큰 브루트포스 방지). |

**향후 확장: 앱(App) 지원 시**

앱 클라이언트는 매직 링크 대신 6자리 OTP 코드 방식을 사용한다. 기존 웹 플로우를 변경하지 않고 아래 엔드포인트를 추가한다.

```
POST /api/v1/auth/password-reset/request/otp   (OTP 코드 이메일 발송)
POST /api/v1/auth/password-reset/verify/otp    (OTP 코드 검증 → 단기 세션 토큰 반환)
PATCH /api/v1/auth/password-reset/otp          (세션 토큰 + 새 비밀번호로 변경)
```

내부적으로 토큰 생성·Redis 저장·이메일 발송 로직을 공유하고, 발송 형식(링크 vs 코드)만 분기 처리한다.

## 5. 비기능 요구사항

### NFR-001: 성능

| 항목 | 기준값 |
|------|--------|
| 로그인/회원가입 응답 시간 (P95) | 200ms 이하 |
| Access Token 갱신 응답 시간 (P95) | 100ms 이하 |
| 비밀번호 재설정 이메일 발송 | 비동기 처리 (응답 시간에 미포함) |

### NFR-002: 보안

| 항목 | 요구사항 |
|------|----------|
| 비밀번호 해싱 | bcrypt, cost factor 12 이상 |
| Refresh Token 저장 | DB에 bcrypt 해시값으로 저장, 평문 미저장 |
| 재설정 토큰 저장 | Redis에 SHA-256 해시값으로 저장 (DB 미저장), TTL 30분 |
| 재설정 페이지 보안 | `Referrer-Policy: no-referrer` 헤더 적용 (토큰 외부 유출 방지) |
| HTTPS | 모든 인증 엔드포인트는 HTTPS 필수 |
| 계정 열거 공격 방지 | 존재하지 않는 이메일에 대한 응답을 성공 응답과 동일하게 처리 |
| Token Rotation | Refresh Token 갱신 시 항상 신규 발급, 이전 토큰 즉시 무효화 |
| Refresh Token 재사용 탐지 | 재사용 시도 시 해당 계정의 모든 Refresh Token 무효화 |

### NFR-003: Rate Limiting

Redis 기반 분산 Rate Limiting 적용. 아래 엔드포인트는 전역 기준값(60초당 10,000회)을 오버라이드한다.

| 엔드포인트 | IP 기준 | 이메일 기준 |
|------------|---------|------------|
| `POST /api/v1/auth/sign-in` | 5회/분 | - |
| `POST /api/v1/auth/password-reset/request` | 5회/시간 | 3회/시간 |
| `PATCH /api/v1/auth/password-reset` | 10회/시간 | - |

### NFR-004: 가용성

| 항목 | 기준값 |
|------|--------|
| 인증 서비스 업타임 | 99.9% 이상 |
| Redis 장애 시 | Fallback으로 DB 기반 Rate Limiting 또는 Rate Limiting 비활성화 (서비스 중단 없음) |

### NFR-005: 로깅 및 감사

- 모든 로그인 시도(성공/실패)를 로그에 기록한다.
- 비밀번호 재설정 요청 및 완료 이벤트를 로그에 기록한다.
- 로그에 비밀번호, 토큰 원문이 포함되지 않도록 한다.
- `CustomClsMiddleware`를 통해 요청별 사용자 ID, IP, User-Agent가 CLS에 저장된다.

### 6.2 데이터 모델

**User 테이블 (인증 관련 필드)**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK, 자동 생성 |
| email | VARCHAR(255) | 유니크, NOT NULL |
| password | VARCHAR(255) | bcrypt 해시값, NOT NULL |
| refreshToken | TEXT | Refresh Token bcrypt 해시값, nullable |
| passwordResetToken | VARCHAR(255) | 재설정 토큰 SHA-256 해시값, nullable |
| passwordResetTokenExpiresAt | TIMESTAMP | 재설정 토큰 만료 시각, nullable |
| status | ENUM | ACTIVE / INACTIVE / WITHDRAWN |
| createdAt | TIMESTAMP | 생성 일시 |
| createdBy | UUID | 생성자 (CLS 자동 주입) |
| updatedAt | TIMESTAMP | 수정 일시 |
| updatedBy | UUID | 수정자 (CLS 자동 주입) |
| isDeleted | BOOLEAN | Soft Delete 여부, 기본값 false |
| deletedAt | TIMESTAMP | 삭제 일시, nullable |
| deletedBy | UUID | 삭제자, nullable |

### 6.3 통합 요구사항

| 시스템 | 목적 | 비고 |
|--------|------|------|
| Redis | Rate Limiting, (선택) 블랙리스트 토큰 관리 | `@libs/common/redis` 모듈 사용 |
| 이메일 서비스 | 비밀번호 재설정 링크 발송 | 비동기 처리 권장 |
| `@libs/common` | 가드, 전략, 공통 예외처리 | `JwtAccessGuard`, `JwtRefreshGuard`, `TokenService` |

### 6.4 아키텍처 결정 사항

- `JwtAccessGuard`는 `APP_GUARD`로 전역 등록되어 있으므로, 인증이 불필요한 Auth 엔드포인트에는 반드시 `@Public()` 데코레이터를 적용한다.
- `JwtRefreshGuard`는 `/auth/refresh` 엔드포인트에만 선택적으로 적용한다.
- 응답 직렬화는 `plainToInstance(ResponseDto, entity, { excludeExtraneousValues: true })` 패턴을 사용하여 `password`, `refreshToken` 등 민감 필드가 응답에 포함되지 않도록 한다.
- 탈퇴 사용자는 물리 삭제 없이 `status=WITHDRAWN`, `isDeleted=true`로 처리한다(Soft Delete).

### 6.5 에러 코드 정의

| 에러 코드 | HTTP 상태 | 설명 |
|-----------|----------|------|
| AUTH_ERROR_001 | 409 Conflict | 이미 가입된 이메일 |
| AUTH_ERROR_002 | 401 Unauthorized | 이메일 또는 비밀번호 불일치 |
| AUTH_ERROR_003 | 401 Unauthorized | 비활성화 또는 탈퇴 계정 |
| AUTH_ERROR_004 | 401 Unauthorized | 만료된 Refresh Token |
| AUTH_ERROR_005 | 401 Unauthorized | 유효하지 않거나 재사용된 Refresh Token |
| AUTH_ERROR_006 | 400 Bad Request | 만료되거나 사용된 비밀번호 재설정 토큰 |

## 7. 의존성 및 리스크

### 7.1 의존성

| 의존성 | 설명 |
|--------|------|
| `@libs/common` | `JwtAccessGuard`, `JwtRefreshGuard`, `TokenService`, `CryptoService` 제공 |
| `@libs/prisma` | `PrismaService` 및 `User` 모델 제공 |
| Redis | Rate Limiting 동작에 필요. 장애 시 Fallback 정책 필요 |
| 이메일 서비스 | 비밀번호 재설정 기능 동작에 필요 |

### 7.2 리스크

| 리스크 | 가능성 | 영향도 | 완화 전략 |
|--------|--------|--------|-----------|
| Refresh Token 탈취 | 중 | 높음 | Token Rotation + 재사용 탐지 + HTTPS 강제 |
| bcrypt 연산으로 인한 로그인 지연 | 중 | 중간 | cost factor 조정(최소 10, 권장 12), P95 성능 모니터링 |
| 이메일 서비스 장애 | 중 | 중간 | 재시도 큐(Bull 등) 적용, 발송 실패 알림 |
| Redis 장애로 Rate Limiting 비활성화 | 저 | 중간 | Redis 이중화 구성 또는 Fallback DB Rate Limiting |
| 계정 열거 공격 | 고 | 중간 | 존재하지 않는 이메일에 동일 응답 반환 |

### 7.3 가정사항

- 클라이언트는 HTTPS로만 서비스에 접근한다.
- Refresh Token은 클라이언트 측에서 HttpOnly 쿠키 또는 안전한 스토리지에 저장한다고 가정한다.
- 이메일 발송 서비스는 별도 인프라로 구성된다.

## 10. 부록

### 10.1 용어 정의

| 용어 | 정의 |
|------|------|
| Access Token | 보호된 리소스에 접근하기 위한 단기 JWT (TTL: 1시간) |
| Refresh Token | Access Token 갱신에 사용되는 장기 JWT (TTL: 7일) |
| Token Rotation | Refresh Token 사용 시마다 새 토큰을 발급하고 이전 토큰을 무효화하는 보안 패턴 |
| Soft Delete | 물리적 레코드 삭제 없이 `isDeleted=true`로 논리 삭제 처리하는 패턴 |
| JTI (JWT ID) | 토큰 고유 식별자. 재사용 탐지에 활용 |
| CLS | Context-Local Storage. 요청 스코프의 컨텍스트 데이터(사용자 ID 등)를 저장하는 메커니즘 |
| Rate Limiting | 단위 시간 내 허용 요청 수를 제한하여 무차별 대입 공격을 방지하는 메커니즘 |

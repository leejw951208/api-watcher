## 1. 배경 및 목적

### 1.1 문제 정의

사용자가 등록한 외부 API의 장애를 실시간에 가깝게 감지하려면 주기적인 자동 호출이 필요하다. 현재 api-watcher 서비스는 모니터 설정(Monitor) 등록 기능을 제공하지만, 등록된 모니터를 자동으로 실행하고 결과를 저장하는 백그라운드 스케줄러가 없어 핵심 가치(API 상태 감시)를 제공하지 못하고 있다.

### 1.2 비즈니스 목표

| # | 목표 | 기대 효과 |
|---|------|-----------|
| BG-1 | API 장애를 사람이 개입하지 않고 자동으로 감지 | 평균 장애 탐지 시간(MTTD) 단축 |
| BG-2 | 호출 결과 이력 보존 | 서비스 품질 추이 분석 및 SLA 대응 자료 확보 |
| BG-3 | 연속 실패 감지 후 알림 파이프라인 연계 | 담당자의 즉각적인 대응 가능 |

### 1.3 성공 지표(KPI)

| 지표 | 목표값 |
|------|--------|
| 스케줄 실행 정확도 | 설정 주기 대비 오차 ±5초 이내 |
| 호출 결과 저장 누락률 | 0% (장애 시 재처리 포함) |
| 연속 실패 알림 트리거 지연 | 마지막 실패 감지 후 10초 이내 |
| 스케줄러 자체 가용성 | 99.9% (월 기준 약 44분 이하 다운타임) |

## 3. 제품 범위

### 3.1 포함 범위 (In Scope)

- 활성화된 모니터 목록을 조회하여 스케줄 등록/갱신
- HTTP 요청 실행 (GET / POST / PUT / PATCH / DELETE, 헤더/바디/인증 토큰 포함)
- 오류 감지: 예상 상태 코드 불일치, 응답 시간 초과, 네트워크 오류
- 호출 결과 로그 저장 (성공/실패, 응답 시간, 상태 코드, 응답 바디)
- 연속 실패 횟수 추적 및 알림 이벤트 발행
- 모니터 비활성화 시 스케줄 즉시 제거

### 3.2 제외 범위 (Out of Scope)

- 알림 채널(이메일, Slack, Webhook) 실제 발송 구현 — 알림 이벤트 발행까지만 담당
- 모니터 CRUD API — 별도 모니터 관리 기능에서 담당
- 호출 결과 조회 API — 별도 로그 조회 기능에서 담당
- 스케줄러 실행 결과에 대한 실시간 대시보드 UI

### 3.3 향후 고려사항

- 분산 환경에서의 중복 실행 방지 (Distributed Lock 도입)
- 동적 스케줄 주기 변경 시 무중단 반영
- 재시도(Retry) 정책 세분화 (지수 백오프)
- 호출 결과 로그 TTL 기반 자동 정리(Archiving)

## 4. 기능 요구사항

### FR-001: 스케줄 초기화 및 동기화

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-001 |
| **기능명** | 스케줄 초기화 및 동기화 |
| **우선순위** | P0 |

**설명**

서버 기동 시 `SchedulerInitService`가 `OnApplicationBootstrap` 훅에서 실행된다. PostgreSQL에서 `isActive = true`인 모든 Monitor 레코드를 조회하고, 각 모니터의 `intervalSeconds` 값을 기반으로 동적 Interval 스케줄을 등록한다.

**수용 기준**

```
Given  서버가 기동될 때
When   SchedulerInitService.onApplicationBootstrap()이 실행되면
Then   isActive = true인 모든 모니터에 대해 intervalSeconds 주기의 동적 스케줄이 등록된다.

Given  이미 스케줄이 등록된 모니터가 존재하는 상태에서
When   동일 monitorId로 재등록 시도가 발생하면
Then   기존 스케줄을 제거하고 새 스케줄로 교체한다 (중복 등록 방지).
```

**기술 상세**

- `@nestjs/schedule`의 `SchedulerRegistry.addInterval()` 사용
- 스케줄 키: `monitor:{monitorId}` 형식으로 일관성 유지
- 기동 시 DB 조회 실패는 예외를 던지지 않고 경고 로그 기록 후 진행 (부분 실패 허용)

### FR-002: HTTP 요청 실행

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-002 |
| **기능명** | HTTP 요청 실행 |
| **우선순위** | P0 |

**설명**

스케줄러 콜백에서 `MonitorExecutorService`가 호출된다. Monitor 설정에 따라 HTTP 요청을 구성하고 실행한다. 응답 수신 또는 오류 발생 시 결과를 반환한다.

**지원 요청 항목**

| 항목 | 설명 |
|------|------|
| HTTP 메서드 | GET, POST, PUT, PATCH, DELETE |
| URL | 모니터에 저장된 `targetUrl` |
| 요청 헤더 | `headers` 필드 (JSON, key-value 형식) |
| 요청 바디 | `body` 필드 (JSON, POST/PUT/PATCH에만 적용) |
| 인증 토큰 | `authToken` 필드가 있으면 `Authorization: Bearer {token}` 헤더 자동 주입 |
| 타임아웃 | `timeoutMs` 필드 (기본값 5,000ms, 최대 30,000ms) |

**수용 기준**

```
Given  Monitor의 method = POST, body = {"key": "value"}, authToken = "abc"인 경우
When   스케줄러가 실행되면
Then   Content-Type: application/json, Authorization: Bearer abc 헤더가 포함된 POST 요청이 전송된다.

Given  타임아웃 설정이 3,000ms이고 대상 서버가 5,000ms 후에 응답하는 경우
When   HTTP 요청이 실행되면
Then   3,000ms 경과 후 타임아웃 오류로 처리되고 결과가 실패로 기록된다.
```

**기술 상세**

- Node.js 내장 `fetch` API 또는 `axios` 사용 (모노레포 내 라이브러리 추가 최소화 우선)
- `AbortController`를 활용한 타임아웃 제어
- SSL 인증서 검증 실패는 별도 설정 필드(`sslVerify`)로 제어 가능하도록 설계 (향후 구현)

### FR-003: 오류 감지 로직

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-003 |
| **기능명** | 오류 감지 로직 |
| **우선순위** | P0 |

**설명**

HTTP 응답 또는 오류를 수신한 후 `MonitorCheckService`가 아래 기준에 따라 성공/실패를 판정한다. 판정 결과는 `CheckStatus` 열거형으로 표현된다.

**오류 유형 분류**

| 오류 유형 | 조건 | CheckStatus |
|-----------|------|-------------|
| 상태 코드 불일치 | 응답 상태 코드가 Monitor의 `expectedStatusCode` 값과 다를 때 | `STATUS_MISMATCH` |
| 응답 시간 초과 | 응답 시간이 Monitor의 `timeoutMs`를 초과할 때 | `TIMEOUT` |
| 네트워크 오류 | DNS 조회 실패, 연결 거부, TLS 핸드셰이크 실패 등 | `NETWORK_ERROR` |
| 성공 | 위 조건 미해당 + 상태 코드 일치 | `SUCCESS` |

**수용 기준**

```
Given  expectedStatusCode = 200이고 실제 응답 상태 코드가 503인 경우
When   MonitorCheckService가 결과를 판정하면
Then   CheckStatus = STATUS_MISMATCH로 결정된다.

Given  네트워크 연결이 불가능한 URL로 요청을 보낸 경우
When   HTTP 요청이 실행되면
Then   CheckStatus = NETWORK_ERROR로 결정되고 errorMessage에 원인이 기록된다.

Given  응답이 정상 수신되고 상태 코드가 일치하는 경우
When   MonitorCheckService가 결과를 판정하면
Then   CheckStatus = SUCCESS로 결정된다.
```

### FR-004: 호출 결과 로그 저장

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-004 |
| **기능명** | 호출 결과 로그 저장 |
| **우선순위** | P0 |

**설명**

매 호출 후 `MonitorLogRepository`가 `MonitorLog` 테이블에 결과를 저장한다. 저장 실패 시 스케줄러 동작을 중단하지 않고 에러 로그만 기록한다.

**MonitorLog 데이터 모델**

```
model MonitorLog {
    id              Int          @id @default(autoincrement())
    monitorId       Int          @map("monitor_id")
    monitor         Monitor      @relation(fields: [monitorId], references: [id])
    status          CheckStatus  @map("status")        // SUCCESS | STATUS_MISMATCH | TIMEOUT | NETWORK_ERROR
    statusCode      Int?         @map("status_code")   // 실제 수신한 HTTP 상태 코드 (네트워크 오류 시 null)
    responseTimeMs  Int          @map("response_time_ms") // 응답 시간 (ms)
    responseBody    String?      @map("response_body") // 응답 바디 (최대 10,000자 자름)
    errorMessage    String?      @map("error_message") // 실패 원인 메시지
    checkedAt       DateTime     @default(now()) @map("checked_at")

    // 공통 감사 필드
    createdAt  DateTime  @default(now()) @map("created_at")
    createdBy  Int?      @map("created_by")
    updatedAt  DateTime? @map("updated_at")
    updatedBy  Int?      @map("updated_by")
    isDeleted  Boolean?  @default(false) @map("is_deleted")
    deletedAt  DateTime? @map("deleted_at")
    deletedBy  Int?      @map("deleted_by")

    @@map("monitor_log")
    @@schema("main")
}
```

**수용 기준**

```
Given  HTTP 호출이 완료(성공 또는 실패)된 경우
When   결과가 저장되면
Then   status, statusCode, responseTimeMs, checkedAt이 모두 저장된다.

Given  응답 바디가 10,000자를 초과하는 경우
When   로그가 저장되면
Then   responseBody는 10,000자로 자려서 저장된다.

Given  MonitorLog 저장 중 DB 오류가 발생한 경우
When   예외가 발생하면
Then   스케줄러는 중단되지 않고 에러 로그만 기록한다.
```

### FR-005: 연속 실패 횟수 추적 및 알림 트리거

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-005 |
| **기능명** | 연속 실패 횟수 추적 및 알림 트리거 |
| **우선순위** | P0 |

**설명**

`MonitorStateService`가 Redis를 사용하여 모니터별 연속 실패 횟수를 관리한다. 실패가 Monitor의 `alertThreshold` 값에 도달하면 알림 이벤트를 발행한다. 성공 시에는 카운터를 초기화한다.

**Redis 키 설계**

| 키 | 형식 | TTL |
|----|------|-----|
| 연속 실패 카운터 | `monitor:fail_count:{monitorId}` | `intervalSeconds * (alertThreshold + 5)` |
| 알림 발송 여부 플래그 | `monitor:alert_sent:{monitorId}` | `alertThreshold * intervalSeconds * 2` |

알림 발송 여부 플래그는 동일한 장애 구간에서 알림이 중복 발행되는 것을 방지한다.

**수용 기준**

```
Given  연속 실패 횟수가 alertThreshold - 1회인 상태에서
When   추가 실패가 발생하면
Then   연속 실패 횟수가 alertThreshold에 도달하고 알림 이벤트가 발행된다.

Given  알림 이벤트가 이미 발행된 상태에서
When   추가 실패가 계속 발생하면
Then   알림 이벤트가 중복 발행되지 않는다 (alert_sent 플래그 확인).

Given  연속 실패 후 성공이 발생한 경우
When   성공 결과가 처리되면
Then   fail_count와 alert_sent 플래그가 모두 초기화된다.
```

**알림 이벤트 페이로드**

```typescript
interface MonitorAlertEvent {
    monitorId: number
    monitorName: string
    targetUrl: string
    consecutiveFailCount: number
    lastCheckStatus: CheckStatus
    lastErrorMessage: string | null
    triggeredAt: Date
}
```

### FR-006: 모니터 비활성화 시 스케줄 중단

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-006 |
| **기능명** | 모니터 비활성화 시 스케줄 중단 |
| **우선순위** | P0 |

**설명**

Monitor의 `isActive` 값이 `false`로 변경되면, 해당 이벤트를 수신한 `SchedulerManagerService`가 등록된 스케줄을 즉시 제거한다. 반대로 `isActive`가 `true`로 변경되면 스케줄을 새로 등록한다.

**수용 기준**

```
Given  isActive = true 상태의 모니터 스케줄이 실행 중인 경우
When   isActive = false로 업데이트 이벤트가 발생하면
Then   해당 모니터의 스케줄이 SchedulerRegistry에서 제거되고 더 이상 실행되지 않는다.

Given  isActive = false 상태의 모니터에 대해
When   isActive = true로 업데이트 이벤트가 발생하면
Then   해당 모니터의 스케줄이 새로 등록되고 intervalSeconds 주기로 실행된다.

Given  스케줄 제거 대상인 monitorId가 SchedulerRegistry에 존재하지 않는 경우
When   제거가 시도되면
Then   예외를 발생시키지 않고 경고 로그만 기록한다.
```

**구현 방식**

모니터 관리 서비스에서 `isActive` 변경 시 NestJS `EventEmitter2`를 통해 `monitor.activated` / `monitor.deactivated` 이벤트를 발행하고, `SchedulerManagerService`가 `@OnEvent()` 데코레이터로 이를 구독한다.

### FR-007: 스케줄 실행 동시성 제어

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-007 |
| **기능명** | 스케줄 실행 동시성 제어 |
| **우선순위** | P1 |

**설명**

동일 모니터에 대해 이전 실행이 완료되기 전에 다음 스케줄이 트리거되는 경우(실행 시간 > 주기) 중복 실행을 방지한다.

**수용 기준**

```
Given  intervalSeconds = 10이고 이전 HTTP 요청이 15초째 실행 중인 경우
When   다음 스케줄 트리거(10초)가 발생하면
Then   이전 실행이 진행 중임을 감지하고 해당 회차 실행을 건너뛴다 (skip 로그 기록).
```

**구현 방식**

Redis에 `monitor:running:{monitorId}` 키를 실행 시작 시 SET(EX = timeoutMs + 1000ms)하고, 실행 완료 시 DEL한다. 다음 트리거 시 키 존재 여부로 실행 중 여부를 판단한다.

## 5. 비기능 요구사항

### NFR-001: 성능

| ID | 요구사항 | 기준값 |
|----|----------|--------|

| NFR-001-2 | 호출 결과 DB 저장 시간 | 200ms 이하 |
| NFR-001-3 | 동시 실행 가능한 모니터 수 | 최소 500개 (Node.js 이벤트 루프 기반 비동기 처리) |

### NFR-002: 보안

| ID | 요구사항 |
|----|----------|
| NFR-002-1 | `authToken` 필드는 DB 저장 시 암호화(AES-256-GCM) 적용. 기존 `CryptoService` 활용 |
| NFR-002-2 | 호출 결과 로그의 `responseBody`에 민감 정보(토큰, 패스워드)가 포함된 경우를 대비해 로그 마스킹 정책 수립 (v1.1 이후 검토) |
| NFR-002-3 | 스케줄러 내부 서비스 간 통신은 동일 프로세스 내 모듈 호출로 제한 (외부 노출 없음) |

### NFR-003: 확장성

| ID | 요구사항 |
|----|----------|
| NFR-003-1 | 모니터 수 증가에 따른 선형적 확장 가능 구조. 단일 프로세스 한계 도달 시 Bull Queue 기반으로 마이그레이션 가능한 인터페이스 설계 |
| NFR-003-2 | `MonitorExecutorService`는 외부 의존성 없이 독립적으로 교체 가능하도록 인터페이스 추상화 |

### NFR-004: 가용성

| ID | 요구사항 |
|----|----------|
| NFR-004-1 | 스케줄러 서비스 재시작 시 `OnApplicationBootstrap`에서 자동으로 모든 스케줄 재등록 |
| NFR-004-2 | Redis 연결 장애 시 연속 실패 카운터 기능은 비활성화되나 HTTP 호출 및 로그 저장은 계속 동작 |

### NFR-005: 관찰 가능성

| ID | 요구사항 |

## 6. 기술 요구사항 및 제약사항

### 6.1 기술 스택

| 분류 | 기술 | 비고 |
|------|------|------|
| 프레임워크 | NestJS 11, TypeScript 5.7 | 기존 모노레포와 동일 |
| 스케줄러 | `@nestjs/schedule` (node-cron 기반) | Bull Queue는 v2.0에서 검토 |
| HTTP 클라이언트 | Node.js 내장 `fetch` + `AbortController` | 외부 라이브러리 추가 없음 |
| 이벤트 버스 | `@nestjs/event-emitter` (EventEmitter2) | 모듈 간 느슨한 결합 |
| DB | PostgreSQL + Prisma 7.2 | 기존 `@libs/prisma` 사용 |
| 캐시/상태 | Redis (ioredis) | 기존 `@libs/common/redis` 모듈 사용 |

### 6.2 모듈 구조 (예시)

```
apps/
  user/src/
    monitor/

        scheduler-manager.service.ts # 스케줄 등록/제거 관리
      executor/
        monitor-executor.service.ts # HTTP 요청 실행
      check/
        monitor-check.service.ts    # 오류 감지 판정
      state/
        monitor-state.service.ts    # 연속 실패 카운터 (Redis)
      log/
        monitor-log.repository.ts   # 결과 로그 저장
```

### 6.3 신규 Prisma 모델

**Monitor 모델 (신규 또는 기존 확장)**

```
model Monitor {
    id                  Int      @id @default(autoincrement())
    userId              Int      @map("user_id")
    name                String   @map("name")
    targetUrl           String   @map("target_url")
    method              HttpMethod @map("method")    // GET|POST|PUT|PATCH|DELETE
    headers             Json?    @map("headers")
    body                Json?    @map("body")
    authToken           String?  @map("auth_token")  // 암호화 저장
    expectedStatusCode  Int      @default(200) @map("expected_status_code")
    intervalSeconds     Int      @map("interval_seconds") // 최소 60, 최대 86400
    timeoutMs           Int      @default(5000) @map("timeout_ms")
    alertThreshold      Int      @default(3) @map("alert_threshold")
    isActive            Boolean  @default(true) @map("is_active")

    // 공통 감사 필드
    createdAt  DateTime  @default(now()) @map("created_at")
    createdBy  Int?      @map("created_by")
    updatedAt  DateTime? @map("updated_at")
    updatedBy  Int?      @map("updated_by")
    isDeleted  Boolean?  @default(false) @map("is_deleted")
    deletedAt  DateTime? @map("deleted_at")
    deletedBy  Int?      @map("deleted_by")

    logs MonitorLog[]

    @@map("monitor")
    @@schema("main")
}

enum HttpMethod {
    GET
    POST
    PUT
    PATCH
    DELETE
    @@schema("main")
}

enum CheckStatus {
    SUCCESS
    STATUS_MISMATCH
    TIMEOUT
    NETWORK_ERROR
    @@schema("main")
}
```

### 6.4 통합 요구사항

| 시스템 | 통합 방식 | 목적 |
|--------|-----------|------|
| 알림 서비스 | EventEmitter2 이벤트 (`monitor.alert`) | 알림 발송 파이프라인 연계 |
| 모니터 관리 API | EventEmitter2 이벤트 (`monitor.activated`, `monitor.deactivated`) | 스케줄 동적 제어 |
| Redis | ioredis (기존 `@libs/common/redis`) | 카운터, 실행 중 플래그 관리 |
| PostgreSQL | PrismaService (기존 `@libs/prisma`) | 모니터 설정 조회, 로그 저장 |

### 6.5 기술적 제약사항

| 제약 | 내용 |
|------|------|
| 스케줄 최소 주기 | 60초 (1분). 과도한 외부 API 호출 방지 |
| 타임아웃 최대값 | 30,000ms. 이벤트 루프 블로킹 방지 |
| 응답 바디 저장 한도 | 10,000자 초과 시 자름 (DB 용량 관리) |
| 스케줄러 배치 대상 | User App (`apps/user`)에 통합. Admin App은 모니터 관리 기능만 담당 |
| CLS 컨텍스트 | 스케줄러는 HTTP 요청 컨텍스트 외부에서 실행되므로 CLS 미적용. 감사 필드 `createdBy`는 null 허용 |

### 7.1 의존성

| 의존성 | 유형 | 담당 |
|--------|------|------|
| Monitor 관리 기능 (CRUD) | 내부 기능 | 백엔드 개발팀 |
| `@nestjs/schedule` 패키지 추가 | 인프라 | 백엔드 개발팀 |
| `@nestjs/event-emitter` 패키지 추가 | 인프라 | 백엔드 개발팀 |
| Redis 인스턴스 (기존) | 외부 인프라 | 인프라팀 |
| PostgreSQL 신규 테이블 마이그레이션 | DB | 백엔드 개발팀 |

### 7.2 리스크

| 리스크 | 가능성 | 영향도 | 완화 전략 |
|--------|--------|--------|-----------|
| 모니터 수 급증으로 단일 프로세스 한계 도달 | 중 | 상 | intervalSeconds 최소값 제한(60초) + 추후 Bull Queue 전환 인터페이스 설계 |
| 대상 서버 느린 응답으로 이벤트 루프 지연 | 중 | 중 | 모든 HTTP 요청을 비동기 처리 + AbortController 타임아웃 강제 |
| Redis 장애 시 연속 실패 알림 미발송 | 저 | 중 | Redis 오류를 catch하고 DB 기반 폴백 로직 검토 (v1.1) |
| 스케줄러 재시작 시 실행 중이던 체크 중단 | 중 | 저 | 재시작 후 스케줄 자동 재등록으로 다음 주기에 재실행됨 (허용 가능한 손실) |
| authToken 암호화 키 관리 실패 | 저 | 상 | 환경변수로 주입, 비밀 관리 서비스(AWS Secrets Manager 등) 연동 검토 |

### 7.3 가정사항

- Monitor 테이블이 이 PRD 구현 전에 존재하거나 동시에 마이그레이션된다.
- Redis는 항상 가용 상태를 유지하거나 장애 시 자동 복구된다.
- 대상 API의 실제 호출이 허용되어 있음(방화벽, IP 화이트리스트 등 사전 협의 완료).
- `@nestjs/schedule`, `@nestjs/event-emitter` 패키지 추가에 대한 팀 합의가 이루어졌다.

## 1. 배경 및 목적

### 1.1 문제 정의

API Watcher는 등록된 API 엔드포인트를 주기적으로 체크하여 장애를 감지한다. 현재는 모니터링 결과를 시스템 내부에만 기록하며, 담당자가 직접 대시보드를 확인해야만 장애를 인지할 수 있다. 이로 인해 장애 인지가 지연되고 서비스 복구 시간(MTTR)이 길어지는 문제가 발생한다.

### 1.2 비즈니스 목표

- 장애 발생 시 담당자에게 실시간으로 알림을 발송하여 MTTR 단축
- 슬랙·이메일 등 팀에서 이미 사용 중인 채널로 알림을 전달하여 별도 도구 설치 부담 제거
- 중복 알림 억제 정책으로 알림 피로도(Alert Fatigue)를 최소화
- 복구 감지 시 정상화 알림을 발송하여 장애 종료를 자동으로 인지

### 1.3 성공 지표 (KPI)

| 지표 | 목표값 |
|------|--------|
| 장애 감지 후 알림 발송까지 소요 시간 | 30초 이내 |
| 알림 발송 실패율 | 1% 미만 |
| 중복 알림 발생 건수 | 0건 (억제 정책 적용 구간 내) |
| 알림 채널 등록 성공률 | 99% 이상 |

### 2.3 이해관계자 요구사항

- **백엔드 개발팀**: 기존 모노레포 구조(`@libs/common`, `@libs/prisma`) 및 코드 컨벤션과 일관성 유지
- **인프라팀**: 외부 서비스(슬랙, 이메일) 호출 실패 시 재시도 및 Dead Letter Queue 처리 방안 필요
- **QA팀**: 알림 발송 이력을 통해 테스트 시나리오 검증 가능해야 함

## 3. 제품 범위

### 3.1 포함 범위 (In Scope)

- 알림 채널(슬랙 Webhook URL, 이메일 주소) CRUD
- 모니터-알림 채널 연결(N:M) 관리
- 장애 감지 시 알림 발송 (슬랙, 이메일)
- 알림 중복 발송 억제 정책 (최초 1회 + 이후 N분 간격)
- API 복구 감지 시 정상화 알림 발송
- 알림 발송 이력 조회 (Offset 페이지네이션)
- Admin API를 통한 전체 알림 채널 및 이력 관리

### 3.2 제외 범위 (Out of Scope)

- SMS, PagerDuty, Webhook(슬랙 외) 등 추가 알림 채널 (v2 이후 검토)
- 알림 발송 스케줄링 UI (대시보드 전용 범위)
- 알림 내용 커스텀 템플릿 편집기
- 이메일 수신 확인(Read Receipt) 기능

### 3.3 향후 고려사항 (v2+)

- SMS, PagerDuty, Microsoft Teams 채널 추가
- 알림 에스컬레이션 정책 (1차 알림 미확인 시 2차 수신자에게 전달)
- 알림 메시지 템플릿 커스터마이징
- 알림 발송 통계 대시보드

## 4. 기능 요구사항

### FR-001. 알림 채널 등록

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-001 |
| **기능명** | 알림 채널 등록 |
| **우선순위** | P0 (필수) |
| **담당 앱** | User App (`POST /api/v1/notification-channels`) |

**설명**

운영자가 슬랙 Webhook URL 또는 이메일 주소를 알림 채널로 등록한다. 채널 유형(`SLACK`, `EMAIL`)에 따라 필수 입력값이 달라진다.

**요청 스펙**

```
POST /api/v1/notification-channels
Authorization: Bearer {access_token}
```

```json
// 슬랙 채널 등록
{
  "type": "SLACK",
  "name": "백엔드 장애 알림",
  "config": {
    "webhookUrl": "https://hooks.slack.com/services/..."
  }
}

// 이메일 채널 등록
{
  "type": "EMAIL",
  "name": "운영팀 이메일",
  "config": {
    "email": "ops@example.com"
  }
}
```

**응답 스펙**

```json
// 201 Created
{
  "data": {
    "id": 1,
    "type": "SLACK",
    "name": "백엔드 장애 알림",
    "config": { "webhookUrl": "https://hooks.slack.com/services/..." },
    "isActive": true,
    "createdAt": "2026-03-19T00:00:00.000Z"
  }
}
```

**수용 기준 (AC)**

- **AC-001-1**: Given 인증된 사용자가 유효한 슬랙 Webhook URL을 입력할 때, When `POST /api/v1/notification-channels`를 호출하면, Then `201 Created`와 함께 생성된 채널 정보가 반환된다.
- **AC-001-2**: Given 인증된 사용자가 유효한 이메일 주소를 입력할 때, When `POST /api/v1/notification-channels`를 호출하면, Then `201 Created`와 함께 생성된 채널 정보가 반환된다.
- **AC-001-3**: Given 슬랙 채널 등록 시 `webhookUrl`이 누락된 경우, When API를 호출하면, Then `400 Bad Request`가 반환된다.
- **AC-001-4**: Given 이메일 채널 등록 시 이메일 형식이 잘못된 경우, When API를 호출하면, Then `400 Bad Request`가 반환된다.
- **AC-001-5**: Given 미인증 요청인 경우, When API를 호출하면, Then `401 Unauthorized`(AUTH_ERROR_001)가 반환된다.

### FR-002. 알림 채널 목록 조회

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-002 |
| **기능명** | 알림 채널 목록 조회 |
| **우선순위** | P0 (필수) |
| **담당 앱** | User App (`GET /api/v1/notification-channels`) |

**설명**

본인이 등록한 알림 채널 목록을 Offset 페이지네이션으로 조회한다.

**요청 스펙**

```
GET /api/v1/notification-channels?page=1&size=20&order=DESC
Authorization: Bearer {access_token}
```

**응답 스펙**

```json
{
  "data": [
    {
      "id": 1,
      "type": "SLACK",
      "name": "백엔드 장애 알림",
      "isActive": true,
      "createdAt": "2026-03-19T00:00:00.000Z"
    }
  ],
  "meta": { "page": 1, "totalCount": 5 }
}
```

**수용 기준 (AC)**

- **AC-002-1**: Given 인증된 사용자가 등록한 채널이 3개 있을 때, When 목록 조회 API를 호출하면, Then 본인 채널만 3개 반환된다.
- **AC-002-2**: Given `isDeleted: true`인 채널은, When 목록 조회 시, Then 결과에 포함되지 않는다.
- **AC-002-3**: Given `config.webhookUrl` 등 민감 정보는, When 목록 조회 응답에, Then 마스킹(`https://hooks.slack.com/services/***`)되어 반환된다.

### FR-004. 알림 채널 수정

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-004 |
| **기능명** | 알림 채널 수정 |
| **우선순위** | P0 (필수) |
| **담당 앱** | User App (`PATCH /api/v1/notification-channels/:id`) |

**설명**

채널 이름, Webhook URL, 이메일 주소, 활성화 여부(`isActive`)를 수정한다. 채널 유형(`type`)은 변경 불가.

**수용 기준 (AC)**

- **AC-004-1**: Given 본인 소유 채널에 유효한 수정 값을 입력하면, Then `200 OK`와 수정된 채널 정보가 반환된다.
- **AC-004-2**: Given `isActive: false`로 수정하면, Then 해당 채널로는 이후 알림이 발송되지 않는다.
- **AC-004-3**: Given `type` 필드를 변경 시도하면, Then `400 Bad Request`가 반환된다.
- **AC-004-4**: Given 타인 소유 채널을 수정하려 하면, Then `403 Forbidden`(NOTIFICATION_ERROR_002)이 반환된다.

### FR-005. 알림 채널 삭제

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-005 |
| **기능명** | 알림 채널 삭제 |
| **우선순위** | P0 (필수) |
| **담당 앱** | User App (`DELETE /api/v1/notification-channels/:id`) |

**설명**

Soft Delete 방식으로 처리한다. 삭제된 채널은 연결된 모니터-채널 연결 레코드에서도 비활성화된다.

**수용 기준 (AC)**

- **AC-005-1**: Given 본인 소유 채널을 삭제하면, Then `200 OK`가 반환되고 `isDeleted: true`로 설정된다.
- **AC-005-2**: Given 이미 삭제된 채널을 재삭제하려 하면, Then `400 Bad Request`(NOTIFICATION_ERROR_005)가 반환된다.
- **AC-005-3**: Given 삭제된 채널은, When 알림 발송 시, Then 해당 채널로는 알림이 발송되지 않는다.

### FR-008. 장애 감지 시 알림 발송

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-008 |
| **기능명** | 장애 감지 시 알림 발송 |
| **우선순위** | P0 (필수) |
| **처리 방식** | 내부 이벤트 기반 (모니터링 체크 완료 후 자동 트리거) |

**설명**

모니터링 체크 결과가 실패(`status: ERROR`)로 판정될 때, 해당 모니터에 연결된 활성 채널 전체로 알림을 발송한다.

**알림 메시지 포함 정보**

| 필드 | 설명 |
|------|------|
| `monitorName` | 모니터 이름 |
| `apiUrl` | 모니터링 대상 API URL |
| `statusCode` | HTTP 응답 상태 코드 |
| `errorMessage` | 오류 상세 메시지 |
| `occurredAt` | 장애 최초 발생 시각 (ISO 8601, KST) |
| `checkId` | 체크 이력 ID (추적용) |

**슬랙 메시지 예시**

```
[장애 감지] 백엔드 서버 헬스체크
API URL: https://api.example.com/health
상태 코드: 503
오류 내용: Service Unavailable
발생 시각: 2026-03-19 09:00:00 KST
```

**수용 기준 (AC)**

- **AC-008-1**: Given 모니터 체크 결과가 ERROR이고 연결된 활성 채널이 있을 때, When 첫 번째 실패가 감지되면, Then 연결된 모든 채널로 장애 알림이 발송된다.
- **AC-008-2**: Given 슬랙 채널이 연결되어 있을 때, Then Incoming Webhook API를 통해 메시지가 발송된다.
- **AC-008-3**: Given 이메일 채널이 연결되어 있을 때, Then 지정된 이메일 주소로 장애 알림 메일이 발송된다.
- **AC-008-4**: Given 알림 발송 성공 시, Then `NotificationLog` 테이블에 `status: SUCCESS`로 이력이 기록된다.
- **AC-008-5**: Given 외부 채널(슬랙/이메일) 호출이 실패한 경우, Then 최대 3회 재시도 후 `NotificationLog`에 `status: FAILED`로 기록된다.

### FR-009. 알림 중복 발송 방지

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-009 |
| **기능명** | 알림 중복 발송 방지 |
| **우선순위** | P0 (필수) |
| **처리 방식** | Redis 기반 억제 키 관리 |

**설명**

연속 장애 상태에서 매 체크마다 알림이 발송되면 운영자의 알림 피로도가 극심해진다. 이를 방지하기 위해 아래 억제 정책을 적용한다.

**억제 정책**

| 상황 | 발송 규칙 |
|------|----------|
| 장애 최초 감지 | 즉시 1회 발송 |
| 연속 장애 지속 중 | 마지막 알림 발송 후 N분이 경과한 경우에만 재발송 (기본값: 60분) |
| 장애 해소 | 정상화 알림 1회 발송 후 억제 키 초기화 |

**억제 키 구조**

```
notification:suppress:{monitorId}:{channelId}
TTL: N분 (설정 가능, 기본 60분)
```

**수용 기준 (AC)**

- **AC-009-1**: Given 최초 장애 감지 후 억제 TTL 이내에 동일 모니터가 재실패하면, When 알림 발송 로직이 실행되어도, Then 해당 채널로 알림이 발송되지 않는다.
- **AC-009-2**: Given 억제 TTL(60분)이 경과한 후 장애가 지속 중이면, When 알림 발송 로직이 실행되면, Then 알림이 재발송되고 억제 키가 갱신된다.
- **AC-009-3**: Given 억제로 인해 발송이 생략된 경우, Then `NotificationLog`에 `status: SUPPRESSED`로 이력이 기록된다.
- **AC-009-4**: Given 모니터별 억제 간격(N분)은, Then 운영자가 모니터 설정에서 변경할 수 있다. (기본값: 60분, 최솟값: 5분)

### FR-010. 복구 감지 시 정상화 알림 발송

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-010 |
| **기능명** | 복구 감지 시 정상화 알림 발송 |
| **우선순위** | P0 (필수) |
| **처리 방식** | 내부 이벤트 기반 (모니터 상태가 ERROR → OK로 전환될 때 트리거) |

**설명**

모니터 상태가 `ERROR`에서 `OK`로 전환될 때, 연결된 모든 활성 채널로 정상화 알림을 발송한다. 발송 후 Redis 억제 키를 즉시 삭제한다.

**정상화 알림 메시지 예시**

```
[장애 해소] 백엔드 서버 헬스체크
API URL: https://api.example.com/health
복구 시각: 2026-03-19 09:45:00 KST
장애 지속 시간: 45분
```

**수용 기준 (AC)**

- **AC-010-1**: Given 모니터 상태가 ERROR에서 OK로 전환되면, Then 연결된 모든 활성 채널로 정상화 알림이 발송된다.
- **AC-010-2**: Given 정상화 알림 발송 후, Then Redis 억제 키가 삭제된다.
- **AC-010-3**: Given 모니터가 ERROR 상태인 적이 없었다면 (억제 키 없음), Then 정상화 알림은 발송되지 않는다.
- **AC-010-4**: Given 정상화 알림 메시지에는, Then 장애 지속 시간(최초 ERROR 감지 시각 ~ 복구 시각)이 포함된다.

## 5. 비기능 요구사항

### NFR-001. 성능

| 항목 | 기준값 |
|------|--------|
| 단건 조회 API | P95 100ms 이하 |
| 목록 조회 API | P95 300ms 이하 |
| 생성·수정 API | P95 200ms 이하 |
| 알림 발송 처리 (내부 로직) | 장애 감지 후 30초 이내 발송 완료 |

### NFR-002. 보안

- 모든 엔드포인트는 `JwtAccessGuard` 전역 적용 (인증 필수)
- Admin API는 추가로 `PermissionGuard` + `@Permission('notification', 'read'|'write'|'delete')` 적용
- `config.webhookUrl`, `config.email` 등 민감 정보는 응답 시 마스킹 처리
- Webhook URL은 DB 저장 시 암호화 고려 (`CryptoService` 활용)

### NFR-003. 신뢰성 및 재시도

- 외부 알림 채널(슬랙 API, 이메일 서비스) 호출 실패 시 지수 백오프(Exponential Backoff)로 최대 3회 재시도
- 3회 재시도 후에도 실패 시 `NotificationLog.status = FAILED`로 기록하고 Dead Letter Queue에 적재
- 알림 발송 실패가 모니터링 체크 로직 자체를 blocking해서는 안 됨 (비동기 분리)

### NFR-004. 확장성

- 알림 발송 로직은 NestJS Event Emitter 또는 BullMQ 큐를 사용하여 비동기 처리
- 향후 채널 유형 추가 시 `NotificationChannel` 전략 패턴으로 확장 가능하도록 설계

### NFR-005. 가용성

- 알림 서비스 장애가 모니터링 서비스 핵심 기능(체크, 이력 저장)에 영향을 주지 않도록 장애 격리
- Redis 억제 키 조회 실패 시 알림 발송을 허용(Fail Open) 방향으로 처리

### NFR-006. 감사 및 추적

- `NotificationLog` 테이블에 모든 발송 시도(성공/실패/억제) 기록
- 공통 감사 필드(`createdAt`, `createdBy`, `updatedAt`, `updatedBy`, `isDeleted`, `deletedAt`, `deletedBy`) 적용

### NFR-007. Rate Limiting

- 알림 채널 등록/수정 엔드포인트: IP + User 기준 5회/분 (민감 작업 오버라이드)
- 알림 이력 조회 엔드포인트: 기본값 유지 (60초당 10,000회)

## 6. 기술 요구사항 및 제약사항

### 6.1 기술 스택

| 분류 | 기술 |
|------|------|
| 프레임워크 | NestJS 11, TypeScript 5.7 |
| ORM | Prisma 7.2 + PostgreSQL 15+ |
| 캐시 (억제 키 관리) | Redis (ioredis) |
| 슬랙 연동 | Slack Incoming Webhook API |
| 이메일 발송 | nodemailer (SMTP) 또는 SendGrid API (v1에서 결정) |
| 비동기 큐 | NestJS EventEmitter (MVP) → BullMQ 전환 고려 (v2) |
| 패키지 관리 | pnpm workspace |

### 6.2 슬랙 연동 상세

- Slack Incoming Webhook URL 방식 사용 (OAuth 불필요, 설정 간단)
- Webhook URL은 채널 등록 시 사용자가 직접 입력
- 발송 실패 시 HTTP 응답 코드 및 body를 `NotificationLog.errorMessage`에 저장

### 6.3 이메일 발송 상세

- **MVP**: nodemailer + SMTP (환경변수로 SMTP 서버 설정)
- **v2 이후**: SendGrid / AWS SES 전환 검토
- 이메일 발신자 주소는 환경변수(`MAIL_FROM`)로 관리
- HTML 템플릿 사용 (장애 / 정상화 각 1개 템플릿)

### 6.4 데이터 모델

#### NotificationChannel (알림 채널)

```prisma
model NotificationChannel {
  id        Int                        @id @default(autoincrement())
  userId    Int
  type      NotificationChannelType    // SLACK | EMAIL
  name      String
  config    Json                       // { webhookUrl } or { email }
  isActive  Boolean                    @default(true)

  // 감사 필드
  createdAt DateTime                   @default(now())
  createdBy Int?
  updatedAt DateTime                   @updatedAt
  updatedBy Int?
  isDeleted Boolean                    @default(false)
  deletedAt DateTime?
  deletedBy Int?

  // 관계
  user      User                       @relation(fields: [userId], references: [id])
  monitors  MonitorNotificationChannel[]
  logs      NotificationLog[]
}

enum NotificationChannelType {
  SLACK
  EMAIL
}
```

#### MonitorNotificationChannel (모니터-채널 연결)

```prisma
model MonitorNotificationChannel {
  id        Int                  @id @default(autoincrement())
  monitorId Int
  channelId Int

  createdAt DateTime             @default(now())
  createdBy Int?
  isDeleted Boolean              @default(false)
  deletedAt DateTime?
  deletedBy Int?

  monitor   Monitor              @relation(fields: [monitorId], references: [id])
  channel   NotificationChannel  @relation(fields: [channelId], references: [id])

  @@unique([monitorId, channelId])
}
```

#### NotificationLog (알림 발송 이력)

```prisma
model NotificationLog {
  id               Int                      @id @default(autoincrement())
  monitorId        Int
  channelId        Int
  notificationType NotificationType         // FAILURE | RECOVERY
  status           NotificationStatus       // SUCCESS | FAILED | SUPPRESSED
  payload          Json                     // 발송한 메시지 내용
  errorMessage     String?
  sentAt           DateTime                 @default(now())

  // 감사 필드
  createdAt        DateTime                 @default(now())
  createdBy        Int?
  updatedAt        DateTime                 @updatedAt
  updatedBy        Int?
  isDeleted        Boolean                  @default(false)
  deletedAt        DateTime?
  deletedBy        Int?

  monitor          Monitor                  @relation(fields: [monitorId], references: [id])
  channel          NotificationChannel      @relation(fields: [channelId], references: [id])
}

enum NotificationType {
  FAILURE
  RECOVERY
}

enum NotificationStatus {
  SUCCESS
  FAILED
  SUPPRESSED
}
```

## 8. 의존성 및 리스크

### 8.1 의존성

| 의존 대상 | 내용 |
|----------|------|
| 모니터링 체크 도메인 | 체크 결과 이벤트 스펙 확정 필요 (`monitorId`, `status`, `statusCode`, `errorMessage`, `checkedAt`) |
| Monitor 모델 | `suppressIntervalMinutes` 필드 추가 필요 |
| Redis 인프라 | 억제 키 관리를 위한 Redis 연결 필수 |
| 외부 SMTP 서버 또는 SendGrid | 이메일 발송을 위한 계정 및 설정 필요 |

### 8.2 리스크

| 리스크 | 가능성 | 영향도 | 완화 전략 |
|--------|--------|--------|----------|
| 슬랙 Webhook URL 유효성 검증 불가 | 중 | 중 | 채널 등록 시 테스트 발송 옵션 제공 (v2) |
| 이메일 발송 서비스 장애 | 중 | 중 | 재시도 + 실패 이력 기록, 향후 복수 발송 수단 확보 |
| Redis 장애 시 억제 키 조회 불가 | 저 | 중 | Fail Open 정책 (억제 키 없으면 발송 허용) |
| 알림 발송 폭주 (다수 모니터 동시 장애) | 중 | 중 | BullMQ 큐 도입으로 발송 속도 제어 (v2) |
| 외부 API Rate Limit 초과 | 저 | 저 | 슬랙/이메일 API Rate Limit 모니터링 |

### 8.3 가정사항

- 모니터링 체크 서비스는 이미 구현되어 있으며, 체크 결과를 `EventEmitter2`로 발행한다.
- `Monitor` 모델에 `suppressIntervalMinutes` 필드를 추가할 수 있다.
- Redis는 이미 `@libs/common/redis`를 통해 공유 인프라로 제공된다.

| 용어 | 정의 |
|------|------|
| 알림 채널(Notification Channel) | 알림을 수신하는 수단. 슬랙 Webhook URL 또는 이메일 주소 |
| 억제 정책(Suppression Policy) | 연속 장애 시 중복 알림을 방지하기 위해 일정 시간 동안 재발송을 막는 정책 |
| 정상화 알림(Recovery Notification) | 장애 해소 시 발송되는 알림 |
| MTTR | Mean Time To Recover. 장애 발생부터 복구까지 평균 소요 시간 |
| Fail Open | 억제 키 조회 실패 시 안전하게 알림을 허용(발송)하는 방향으로 처리하는 정책 |
| Dead Letter Queue (DLQ) | 재시도 후에도 처리 실패한 메시지를 별도 보관하는 큐 |

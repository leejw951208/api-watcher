## 1. 배경 및 목적

### 1.1 문제 정의

api-watcher는 등록된 외부 API를 주기적으로 호출하여 상태를 감시하는 서비스다. 모니터링 스케줄러가 API를 호출한 결과를 저장하고 사용자가 조회할 수 있는 기능이 없으면, 다음과 같은 문제가 발생한다.

- 과거에 어떤 장애가 발생했는지 추적할 수 없다.
- 특정 시간대의 오류를 재현하거나 원인을 분석하기 어렵다.
- 업타임이나 응답 시간 변화를 수치로 파악하기 어렵다.

### 1.2 비즈니스 목표

- 모니터링 이력을 체계적으로 보관하여 장애 원인 분석에 활용한다.
- 업타임·응답 시간 통계를 통해 서비스 SLA를 정량적으로 측정한다.
- 로그 자동 삭제 정책으로 데이터베이스 용량 비용을 통제한다.

### 1.3 성공 지표

| 지표 | 목표 |
|------|------|
| 로그 단건 조회 응답 시간 | P95 100ms 이하 |
| 로그 목록 조회 응답 시간 | P95 300ms 이하 |
| 통계 조회 응답 시간 | P95 500ms 이하 |
| 로그 보관 자동 삭제 실행 성공률 | 99.9% 이상 |
| 스케줄러 로그 저장 누락률 | 0.1% 미만 |

## 2. 사용자 및 이해관계자

### 2.1 타겟 사용자

| 페르소나 | 설명 |
|---------|------|
| **서비스 운영자** | 등록한 API의 상태를 주기적으로 점검하고, 장애 발생 시 원인을 파악해야 하는 사용자 |
| **개발자** | 자신이 운영하는 API의 응답 시간 추이와 오류 패턴을 분석하는 사용자 |
| **서비스 관리자 (Admin)** | 전체 모니터의 로그 현황을 조회하고, 보관 정책을 설정하는 내부 관리자 |

### 2.2 사용자 스토리

| ID | 사용자 스토리 |
|----|-------------|
| US-001 | 나는 **서비스 운영자**로서, 특정 모니터의 호출 이력을 날짜 범위와 성공/실패 조건으로 필터링하여 조회하고 싶다. 장애 발생 시간대를 빠르게 특정하기 위해서다. |
| US-002 | 나는 **서비스 운영자**로서, 특정 호출 건의 상세 정보(응답 코드, 응답 시간, 응답 바디, 오류 메시지)를 확인하고 싶다. 오류 원인을 정확히 파악하기 위해서다. |
| US-003 | 나는 **개발자**로서, 모니터별 업타임 비율과 평균 응답 시간, 기간별 오류 횟수를 한눈에 파악하고 싶다. API 품질을 지속적으로 개선하기 위해서다. |
| US-004 | 나는 **서비스 운영자**로서, 오류가 발생한 호출 이력만 빠르게 필터링하여 조회하고 싶다. 성공 이력을 제외하고 문제 구간에 집중하기 위해서다. |
| US-005 | 나는 **서비스 관리자**로서, 모니터별로 로그 보관 기간을 설정하고, 기간이 지난 로그를 자동으로 삭제하고 싶다. 데이터베이스 용량을 효율적으로 관리하기 위해서다. |

### 2.3 이해관계자 요구사항

- **백엔드 개발팀**: 스케줄러와 로그 저장 로직이 분리되어 결합도가 낮아야 한다.
- **프론트엔드 개발팀**: 무한 스크롤 UI를 위한 커서 페이지네이션 응답 형식이 필요하다.
- **서비스 운영팀**: 통계 데이터는 실시간이 아닌 근사값(캐시 기반)이어도 무방하다.

## 3. 제품 범위

### 3.1 포함 범위 (In Scope)

- 호출 이력 목록 조회 (모니터별, 날짜 범위, 성공/실패 필터, 커서 페이지네이션)
- 호출 이력 단건 조회
- 모니터별 통계 조회 (업타임 비율, 평균 응답 시간, 기간별 오류 횟수)
- 오류 이력 전용 조회 (목록 형태)
- 로그 보관 기간 설정 및 자동 삭제 (배치 스케줄러)
- Admin App: 전체 모니터 로그 조회 및 보관 정책 관리

### 3.2 제외 범위 (Out of Scope)

- 실시간 스트리밍 로그 조회 (WebSocket 기반)
- 로그 기반 알림 발송 (별도 알림 기능 PRD에서 다룸)
- 외부 로그 수집 시스템(ELK, Datadog 등)과의 연동
- 로그 데이터 CSV/Excel 내보내기

### 3.3 향후 고려사항

- 응답 시간 추이 시계열 그래프 데이터 API
- 로그 이상 탐지(Anomaly Detection) 기반 자동 알림
- 로그 데이터 외부 내보내기 (CSV/Excel)

## 4. 기능 요구사항

### FR-001: 호출 이력 목록 조회

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-001 |
| **기능명** | 호출 이력 목록 조회 |
| **우선순위** | P0 (필수) |
| **대상 앱** | User App (`GET /api/v1/monitors/{monitorId}/logs`) |

**설명**

특정 모니터에 속한 호출 이력을 커서 페이지네이션으로 조회한다. 날짜 범위와 성공/실패 여부로 필터링할 수 있다.

**요청 파라미터**

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| `monitorId` | number (path) | Y | 모니터 ID |
| `lastCursor` | number (query) | N | 마지막으로 조회한 로그 ID (첫 페이지는 생략) |
| `size` | number (query) | N | 페이지 크기 (기본값: 20, 최대: 100) |
| `order` | string (query) | N | 정렬 방향 (`DESC` \| `ASC`, 기본값: `DESC`) |
| `startDate` | string (query) | N | 조회 시작 일시 (ISO 8601, 예: `2026-03-01T00:00:00Z`) |
| `endDate` | string (query) | N | 조회 종료 일시 (ISO 8601) |
| `status` | string (query) | N | 성공/실패 필터 (`SUCCESS` \| `FAILURE`) |

**응답 형식**

```json
{
  "data": [
    {
      "id": 1024,
      "monitorId": 5,
      "statusCode": 200,
      "responseTimeMs": 134,
      "isSuccess": true,
      "errorMessage": null,
      "calledAt": "2026-03-19T10:00:00Z"
    }
  ],
  "meta": {
    "nextCursor": 980
  }
}
```

**수용 기준 (Acceptance Criteria)**

- Given: 인증된 사용자가 자신의 모니터에 대해 목록 조회를 요청할 때
  - When: `monitorId`, `size`, `lastCursor`를 전달하면
  - Then: 해당 모니터의 이력을 `size` 개 단위로 반환하고, 다음 페이지가 있으면 `nextCursor`를 포함한다.
- Given: `status=FAILURE` 필터를 적용하면
  - When: 목록 조회를 요청하면
  - Then: `isSuccess=false`인 로그만 반환된다.
- Given: `startDate`와 `endDate`가 전달되면
  - When: 목록 조회를 요청하면
  - Then: 해당 범위 내의 `calledAt`을 가진 로그만 반환된다.
- Given: 요청 사용자가 해당 모니터의 소유자가 아니면
  - When: 목록 조회를 요청하면
  - Then: `LOG_ERROR_002(403)` 를 반환한다.
- Given: `monitorId`에 해당하는 모니터가 존재하지 않으면
  - When: 목록 조회를 요청하면
  - Then: `LOG_ERROR_001(404)` 를 반환한다.

### FR-002: 호출 이력 단건 조회

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-002 |
| **기능명** | 호출 이력 단건 조회 |
| **우선순위** | P0 (필수) |
| **대상 앱** | User App (`GET /api/v1/monitors/{monitorId}/logs/{logId}`) |

**설명**

특정 호출 이력의 상세 정보를 조회한다. 응답 바디(최대 10KB)와 오류 메시지를 포함한다.

**응답 형식**

```json
{
  "data": {
    "id": 1024,
    "monitorId": 5,
    "statusCode": 500,
    "responseTimeMs": 3200,
    "isSuccess": false,
    "responseBody": "{\"error\": \"Internal Server Error\"}",
    "errorMessage": "HTTP 500 응답 수신",
    "calledAt": "2026-03-19T10:00:00Z"
  }
}
```

**수용 기준 (Acceptance Criteria)**

- Given: 인증된 사용자가 자신의 모니터에 속한 로그를 조회할 때
  - When: 유효한 `logId`를 전달하면
  - Then: 상태 코드, 응답 시간, 응답 바디, 오류 메시지를 포함한 상세 정보를 반환한다.
- Given: `logId`가 존재하지 않으면
  - When: 단건 조회를 요청하면
  - Then: `LOG_ERROR_001(404)` 를 반환한다.
- Given: 요청 사용자가 해당 모니터의 소유자가 아니면
  - When: 단건 조회를 요청하면
  - Then: `LOG_ERROR_002(403)` 를 반환한다.
- Given: 응답 바디가 10KB를 초과하면
  - When: 로그가 저장될 때
  - Then: 10KB까지만 잘라서 저장하고, 나머지는 폐기한다.

### FR-003: 모니터 통계 조회

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-003 |
| **기능명** | 모니터 통계 조회 |
| **우선순위** | P0 (필수) |
| **대상 앱** | User App (`GET /api/v1/monitors/{monitorId}/logs/stats`) |

**설명**

특정 모니터의 집계 통계를 조회한다. 업타임 비율, 평균 응답 시간, 기간별 오류 횟수를 반환한다. 통계는 Redis 캐시를 통해 5분 단위로 갱신된다.

**응답 형식**

```json
{
  "data": {
    "monitorId": 5,
    "uptimePercent": 99.72,
    "avgResponseTimeMs": 187,
    "errorCount": {
      "last24h": 2,
      "last7d": 8,
      "last30d": 21
    },
    "calculatedAt": "2026-03-19T10:05:00Z"
  }
}
```

**수용 기준 (Acceptance Criteria)**

- Given: 인증된 사용자가 자신의 모니터에 대해 통계 조회를 요청할 때
  - When: `monitorId`를 전달하면
  - Then: 업타임 비율(%), 평균 응답 시간(ms), 최근 24시간/7일/30일 오류 횟수를 반환한다.
- Given: Redis 캐시에 통계가 존재하면
  - When: 통계 조회를 요청하면
  - Then: DB 쿼리 없이 캐시된 결과를 반환한다.
- Given: Redis 캐시가 만료되었거나 존재하지 않으면
  - When: 통계 조회를 요청하면
  - Then: DB에서 집계 쿼리를 실행하고 결과를 캐시에 저장(TTL: 5분)한 후 반환한다.
- Given: 해당 모니터에 로그가 하나도 없으면
  - When: 통계 조회를 요청하면
  - Then: `uptimePercent: null`, `avgResponseTimeMs: null`, `errorCount: { last24h: 0, last7d: 0, last30d: 0 }`을 반환한다.
- 업타임 비율 계산식: `(성공 호출 수 / 전체 호출 수) * 100`, 소수점 둘째 자리까지 표시

### FR-004: 오류 이력 전용 조회

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-004 |
| **기능명** | 오류 이력 전용 조회 |
| **우선순위** | P1 (중요) |
| **대상 앱** | User App (`GET /api/v1/monitors/{monitorId}/logs/errors`) |

**설명**

특정 모니터에서 발생한 오류 이력만 조회하는 전용 엔드포인트다. FR-001의 `status=FAILURE` 필터와 동일한 결과를 반환하지만, 오류 분석 UI에서 별도 진입점이 필요하므로 독립된 엔드포인트로 제공한다. 내부적으로 FR-001 로직을 재사용한다.

**요청 파라미터**

FR-001과 동일하되, `status` 파라미터는 고정으로 `FAILURE`가 적용된다.

**수용 기준 (Acceptance Criteria)**

- Given: 인증된 사용자가 오류 이력을 조회할 때
  - When: `GET /monitors/{monitorId}/logs/errors`를 요청하면
  - Then: `isSuccess=false`인 이력만 커서 페이지네이션으로 반환한다.
- Given: `startDate`, `endDate`를 전달하면
  - When: 오류 이력 조회를 요청하면
  - Then: 해당 기간 내 오류 이력만 반환된다.
- 소유권/존재 여부 검증은 FR-001과 동일한 에러 코드를 사용한다.

### FR-005: 로그 자동 저장 (스케줄러 연동)

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-005 |
| **기능명** | 호출 이력 자동 저장 |
| **우선순위** | P0 (필수) |
| **대상 앱** | 내부 스케줄러 (User App 내 `MonitorLogService`) |

**설명**

모니터링 스케줄러가 외부 API를 호출한 후, 그 결과를 `MonitorLog` 테이블에 저장한다. 저장 실패 시 외부 API 호출 자체에 영향을 주지 않는다.

**저장 데이터**

| 필드 | 설명 |
|------|------|
| `monitorId` | 호출한 모니터 ID |
| `statusCode` | HTTP 응답 상태 코드 (타임아웃/네트워크 오류 시 null) |
| `responseTimeMs` | 응답 수신까지 소요 시간(ms) |
| `isSuccess` | 성공 여부 (설정된 기대 상태 코드와 일치 여부) |
| `responseBody` | 응답 바디 (최대 10KB, 초과 시 잘라서 저장) |
| `errorMessage` | 오류 메시지 (타임아웃, 네트워크 오류, 예상치 못한 상태 코드 등) |
| `calledAt` | 호출 시각 |

**수용 기준 (Acceptance Criteria)**

- Given: 스케줄러가 외부 API 호출에 성공하면
  - When: 응답을 수신하면
  - Then: `statusCode`, `responseTimeMs`, `isSuccess=true`, `responseBody`를 기록하고 저장한다.
- Given: 스케줄러가 외부 API 호출에 실패하면(타임아웃, 5xx, 네트워크 오류)
  - When: 오류가 발생하면
  - Then: `isSuccess=false`, `errorMessage`에 오류 내용을 저장한다. `statusCode`는 HTTP 오류면 해당 코드, 네트워크/타임아웃 오류면 null로 저장한다.
- Given: 로그 저장이 실패하면
  - When: DB 저장 중 예외가 발생하면
  - Then: 예외를 catch하여 오류 로그만 기록하고, 모니터 스케줄러 다음 실행에 영향을 주지 않는다.

### FR-006: 로그 보관 기간 설정 및 자동 삭제

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-006 |
| **기능명** | 로그 보관 기간 설정 및 자동 삭제 |
| **우선순위** | P1 (중요) |
| **대상 앱** | User App (설정) + 내부 배치 스케줄러 (삭제 실행) |

**설명**

모니터별로 로그 보관 기간(일 단위)을 설정할 수 있다. 매일 새벽 특정 시간에 배치 스케줄러가 보관 기간이 지난 로그를 물리 삭제한다. 로그 데이터는 소프트 딜리트가 아닌 물리 삭제를 적용한다(용량 관리 목적).

**보관 기간 설정 API**

`PATCH /api/v1/monitors/{monitorId}/log-retention`

```json
// 요청
{ "retentionDays": 30 }

// 응답
{ "data": { "monitorId": 5, "retentionDays": 30 } }
```

**자동 삭제 배치 스케줄러**

- 실행 주기: 매일 새벽 03:00 (KST, Cron: `0 18 * * *` UTC)
- 삭제 조건: `calledAt < NOW() - retentionDays`이고 해당 모니터의 `retentionDays`가 설정된 로그
- 삭제 방식: 물리 삭제 (`DELETE FROM monitor_log WHERE ...`)
- 1회 삭제 최대 건수: 10,000건 (성능 보호). 초과 시 다음 실행에 이어서 처리한다.

**수용 기준 (Acceptance Criteria)**

- Given: 사용자가 `retentionDays: 30`을 설정하면
  - When: `PATCH /monitors/{monitorId}/log-retention`을 요청하면
  - Then: 해당 모니터의 보관 기간이 30일로 저장된다.
- Given: `retentionDays`가 설정된 모니터에 만료 로그가 존재할 때
  - When: 배치 스케줄러가 실행되면
  - Then: 보관 기간이 지난 로그를 물리 삭제하고, 삭제 건수를 애플리케이션 로그에 기록한다.
- Given: `retentionDays`가 설정되지 않은 모니터는
  - When: 배치 스케줄러가 실행되면
  - Then: 해당 모니터의 로그는 삭제하지 않는다(무기한 보관).
- 보관 기간 유효 범위: 최소 1일, 최대 365일. 범위를 벗어나면 `LOG_ERROR_003(400)` 반환.

### FR-007: Admin — 전체 로그 조회 및 보관 정책 관리

| 항목 | 내용 |
|------|------|
| **기능 ID** | FR-007 |
| **기능명** | Admin 전체 로그 조회 및 보관 정책 일괄 설정 |
| **우선순위** | P2 (있으면 좋음) |
| **대상 앱** | Admin App (`/admin/api/v1/monitor-logs`) |
| **필요 권한** | `monitor-log:read`, `monitor-log:update` |

**설명**

관리자는 모든 모니터의 로그를 조회하고, 보관 기간을 일괄 설정할 수 있다.

**수용 기준 (Acceptance Criteria)**

- Given: `monitor-log:read` 권한을 가진 관리자가
  - When: `GET /admin/api/v1/monitor-logs?monitorId=5`를 요청하면
  - Then: 해당 모니터의 로그 목록을 반환한다 (소유권 제한 없음).
- Given: `monitor-log:update` 권한을 가진 관리자가
  - When: 기본 보관 기간을 일괄 설정하면
  - Then: `retentionDays`가 미설정된 모든 모니터에 해당 기간이 적용된다.

## 5. 비기능 요구사항

### NFR-001: 성능

| ID | 요구사항 | 기준값 |
|----|---------|--------|
| NFR-001-1 | 호출 이력 단건 조회 응답 시간 | P95 100ms 이하 |
| NFR-001-2 | 호출 이력 목록 조회 응답 시간 | P95 300ms 이하 |
| NFR-001-3 | 통계 조회 응답 시간 (캐시 히트) | P95 50ms 이하 |
| NFR-001-4 | 통계 조회 응답 시간 (캐시 미스) | P95 500ms 이하 |
| NFR-001-5 | 배치 삭제 1회 실행 최대 소요 시간 | 10분 이내 |

### NFR-002: 보안

- 모든 엔드포인트는 `JwtAccessGuard`가 기본 적용된다.
- 사용자는 자신이 소유한 모니터의 로그만 조회 가능하다. 소유권 검증은 Service 레이어에서 수행한다.
- Admin App의 로그 관련 API는 `@Permission('monitor-log', 'read')` 및 `@Permission('monitor-log', 'update')` 권한이 필요하다.
- 응답 바디에 민감 정보(인증 헤더, API 키 등)가 포함될 수 있으므로, Admin 권한 없이는 `responseBody` 필드를 응답에서 제외한다.

### NFR-003: 확장성

- `monitor_log` 테이블은 데이터가 빠르게 증가하므로, `monitorId + calledAt` 복합 인덱스를 생성하여 범위 조회 성능을 보장한다.
- 커서 페이지네이션을 사용하여 대용량 테이블에서도 일정한 응답 성능을 유지한다.
- 통계 집계는 DB 직접 쿼리 대신 Redis 캐시(TTL: 5분)를 우선 활용한다.

### NFR-004: 가용성

- 배치 삭제 스케줄러 실패 시 서비스 가용성에 영향을 주지 않는다.
- 스케줄러 실행 이력(성공/실패, 삭제 건수)을 애플리케이션 로그에 기록한다.
- Redis 장애 시 통계 조회는 캐시를 건너뛰고 DB에서 직접 조회한다(Fallback).

### NFR-005: 데이터 무결성

- 로그는 Soft Delete가 아닌 물리 삭제를 적용한다 (스토리지 비용 절감 목적).
- 단, 공통 감사 필드(`createdAt`, `createdBy`)는 저장 시 자동 주입된다.
- 로그 데이터는 수정이 불가능하다. 저장 후 변경을 허용하지 않는다.

### NFR-006: 접근성 및 API 문서

- 모든 API는 Swagger로 문서화한다.
- User App Swagger: `GET /api/docs`
- Admin App Swagger: `GET /admin/api/docs`

### 6.2 데이터 모델

**`MonitorLog` 테이블** (`main` 스키마)

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | `BIGINT` | PK, Auto Increment | 로그 ID (대용량을 고려해 BIGINT 사용) |
| `monitorId` | `INT` | FK, NOT NULL | 연결된 모니터 ID |
| `statusCode` | `INT` | NULL 허용 | HTTP 응답 상태 코드 |
| `responseTimeMs` | `INT` | NOT NULL | 응답 시간(ms) |
| `isSuccess` | `BOOLEAN` | NOT NULL | 성공 여부 |
| `responseBody` | `TEXT` | NULL 허용 | 응답 바디 (최대 10KB) |
| `errorMessage` | `TEXT` | NULL 허용 | 오류 메시지 |
| `calledAt` | `TIMESTAMPTZ` | NOT NULL | 호출 시각 |
| `createdAt` | `TIMESTAMPTZ` | NOT NULL, DEFAULT NOW() | 생성 일시 |
| `createdBy` | `INT` | NULL 허용 | 생성자 (CLS 자동 주입) |

**`Monitor` 테이블 추가 컬럼**

| 컬럼 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `retentionDays` | `INT` | NULL | 로그 보관 기간(일). NULL이면 무기한 보관 |

**인덱스**

```sql
-- 모니터별 + 시간 범위 조회 최적화 (목록 조회, 통계 집계 공통)
CREATE INDEX idx_monitor_log_monitor_called ON monitor_log(monitor_id, called_at DESC);

-- 커서 페이지네이션 최적화
CREATE INDEX idx_monitor_log_id ON monitor_log(id DESC);

-- 오류 이력 전용 조회 최적화
CREATE INDEX idx_monitor_log_monitor_success_called ON monitor_log(monitor_id, is_success, called_at DESC);
```

### 6.3 통합 요구사항

- **모니터링 스케줄러**: `MonitorLogService.saveLog()`를 호출하여 결과를 저장한다. 스케줄러와 로그 저장 로직은 의존성 방향이 단방향(스케줄러 → 로그 서비스)이어야 한다.
- **Redis 캐시**: 통계 데이터를 캐시한다. 캐시 키 형식: `monitor:stats:{monitorId}`, TTL: 300초(5분).
- **@libs/common**: 공통 `CursorRequestDto`, `CursorResponseDto`, `BaseException` 재사용.
- **@libs/prisma**: `PrismaService`를 통해서만 DB에 접근한다.

### 6.4 기술적 제약사항

- Prisma ESLint 규칙에 따라 `PrismaClient`를 직접 import하지 않고 `PrismaService`를 통해서만 접근한다.
- 응답 직렬화는 `plainToInstance(ResponseDto, entity, { excludeExtraneousValues: true })` 패턴을 따른다.
- 배치 스케줄러는 `@nestjs/schedule`의 `@Cron` 데코레이터를 사용한다.

## 8. 의존성 및 리스크

### 8.1 의존성

| 의존 대상 | 내용 | 영향 |
|---------|------|------|
| **모니터(Monitor) 기능** | `MonitorLog`는 `Monitor`에 FK로 연결된다. 모니터 기능이 먼저 구현되어야 한다. | 선행 작업 필수 |
| **Redis** | 통계 캐시에 필요. Redis 없이도 DB Fallback으로 동작 가능하다. | 선택적 의존성 |
| **`@nestjs/schedule`** | 배치 삭제 스케줄러에 필요. | 패키지 추가 필요 여부 확인 |

### 8.2 리스크

| 리스크 | 가능성 | 영향 | 완화 전략 |
|--------|--------|------|---------|
| `monitor_log` 테이블 급격한 용량 증가 | 높음 | 높음 | `retentionDays` 기본값을 30일로 권장하고, 배치 삭제 정상 동작 모니터링 |
| 통계 집계 쿼리 성능 저하 | 중간 | 중간 | 복합 인덱스 + Redis 캐시 5분 TTL로 DB 부하 분산 |
| 배치 삭제 중 테이블 락(Lock) 경합 | 낮음 | 높음 | 1회 10,000건 제한, 새벽 트래픽이 낮은 시간대 실행 |
| 응답 바디에 민감 정보 노출 | 중간 | 높음 | `responseBody`는 Admin 권한 보유자에게만 노출 |

### 8.3 가정사항

- `Monitor` 모델에 `userId`(소유자) 필드가 존재하며, 소유권 검증에 활용 가능하다.
- 스케줄러는 단일 인스턴스로 실행된다. 멀티 인스턴스 환경에서는 분산 락(Distributed Lock) 도입이 필요하다.
- 통계 캐시 갱신 주기 5분은 운영팀이 허용하는 근사값 오차 범위다.

### 10.1 용어 정의

| 용어 | 정의 |
|------|------|

| **P95** | 전체 요청 중 하위 95%가 해당 응답 시간 이내에 처리됨을 의미하는 성능 지표 |

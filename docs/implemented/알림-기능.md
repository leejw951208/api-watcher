# 알림 기능

## 개요

Admin App에서 발송, User App에서 수신/읽기. `userId`가 null이면 전체 사용자 대상 공지. 읽음 상태는 별도 `NotificationRead` 테이블로 관리.

## Admin: 알림 발송/관리

| 엔드포인트 | 설명 | Permission |
|-----------|------|------------|
| `POST /notifications` | 알림 생성 (개별 또는 전체) | `notification:write` |
| `GET /notifications` | 알림 목록 (Offset) | `notification:read` |

**검색 조건**: `userId?`, `type?`, `keyword?` (제목+내용 검색)

생성 시 `userId`를 지정하면 개인 알림, 생략하면 전체 공지로 생성.

## User: 알림 수신

| 엔드포인트 | 설명 |
|-----------|------|
| `GET /notifications/me/cursor` | 내 알림 목록 (Cursor) |
| `GET /notifications/:id` | 알림 상세 |
| `PATCH /notifications/:id` | 읽음 처리 |
| `PATCH /notifications/read-all` | 전체 읽음 처리 |
| `DELETE /notifications/:id` | Soft Delete |

**필터**: `isRead?`, `type?` (SYSTEM, POST, COMMENT, USER)

### 알림 조회 쿼리

```typescript
where: {
    isDeleted: false,
    OR: [{ userId }, { userId: null }],   // 개인 알림 + 전체 공지
    ...(isRead === true && { notificationReads: { some: { userId } } }),
    ...(isRead === false && { notificationReads: { none: { userId } } })
}
```

### 읽음 상태 매핑

알림 목록 조회 시 2번의 쿼리로 N+1 방지:
1. 알림 목록 조회
2. 해당 알림들의 읽음 기록 배치 조회 (`findReadsByNotificationIds`)
3. `Map`으로 O(1) 매핑하여 `isRead`, `readAt` 필드 추가

### 전체 읽음 처리

1. 읽지 않은 알림 ID 목록 조회 (`notificationReads: { none: { userId } }`)
2. `createMany`로 일괄 읽음 기록 생성

### 단건 읽음 처리

이미 읽은 알림이면 조기 반환 (멱등성 보장).

## 알림 타입

| 타입 | 설명 |
|------|------|
| SYSTEM | 시스템 공지 |
| POST | 게시글 관련 |
| COMMENT | 댓글 관련 |
| USER | 사용자 관련 |
